## 1. 執行摘要

### 1.1 產品願景
建立高端包養約會平台，通過誠信分數系統和多重驗證機制，為用戶提供安全、真實、高品質的交友體驗。

### 1.2 產品使命
**"讓每一次相遇都值得信賴"** - 透過科技手段消除網路交友的不信任感，建立真實可靠的交友環境。

### 1.3 核心價值主張
- **安全第一**：多重身份驗證確保用戶真實性
- **誠信為本**：創新誠信分數系統建立信任機制
- **品質優先**：嚴格審核機制保障用戶品質
- **隱私保護**：業界領先的隱私保護措施

## 2. 市場分析與產品定位

### 2.1 目標市場
- **市場規模**：台灣線上交友市場約30億台幣，年增長率15%
- **目標族群**：25-45歲高收入專業人士
- **市場空白**：高端用戶缺乏可信任的交友平台

## 3. 目標用戶分析

### 3.1 主要用戶畫像

#### 用戶畫像A：職業女性「品質追求者」
- **基本資料**：25-35歲，大學以上學歷，月收入5萬+
- **職業背景**：金融、醫療、教育、科技業專業人士
- **使用動機**：尋找穩定關係，重視對方經濟能力和人品
- **痛點**：害怕遇到不實用戶、擔心隱私外洩、時間有限
- **需求**：高效篩選、安全保障、品質保證

#### 用戶畫像B：成功男士「時間寶貴者」
- **基本資料**：30-45歲，事業有成，月收入10萬+
- **職業背景**：企業主管、創業家、專業服務業
- **使用動機**：擴展社交圈，尋找高品質伴侶
- **痛點**：時間有限、假資料太多、低效溝通
- **需求**：精準配對、真實驗證、高效率

<font color=red>經紀先不考慮</font>
#### 用戶畫像C：經紀公司「專業服務者」
- **基本資料**：經紀公司代表，管理多位甜心
- **服務模式**：提供專業經紀服務，確保服務品質
- **使用動機**：為旗下甜心尋找優質甜爹
- **痛點**：平台管理複雜、缺乏專業工具
- **需求**：多帳號管理、專業功能、收益分成

### 3.2 用戶分級體系

### 3.2 用戶分級體系

> **設計原則（v1.2 更新）**：  
> 1. 新增「驗證女會員 (Level 1.5)」作為女性專屬進階驗證層級，與原男性進階驗證（Level 2）分離。  
> 2. 每個層級可開放的功能改為**後台勾選制**，管理員可在後台「會員等級功能設定」介面，針對各層級自由勾選/取消勾選功能項目，無需修改程式碼即可調整策略。

```
註冊會員 (Level 0) - 完成 Email 驗證後取得
│  預設功能（可後台調整）：
├─ 瀏覽探索列表
├─ 有限搜尋（基礎篩選）
└─ 每日聊天 5 則

驗證會員 (Level 1) - 手機 SMS 驗證後取得（不分性別）
│  預設功能（可後台調整，累加 Level 0）：
├─ 完整個人資料編輯
├─ 進階搜尋
├─ 每日聊天 30 則
└─ 進階驗證入口（男性信用卡 / 女性自拍照）

   設計意圖：Email 驗證為註冊前置流程（驗證後才能進入手機驗證階段），
   因此到達 Lv1 升級判斷時 email_verified=true 必然為前置條件。程式碼
   實作（AuthController::verifyPhone、User::getBaseMembershipLevel）只
   檢查 phone_verified，不重複檢查 email_verified。production 統計顯示
   0 個「phone_verified=true 但 email_verified=false」異常 user，證明此
   前置流程設計已落實。

驗證女會員 (Level 1.5) - 女性完成真人照片驗證後取得
│  預設功能（可後台調整，累加 Level 1）：
├─ 查看對方完整個人資料
├─ 每日聊天 100 則（或無上限，後台可設定）
└─ 發送動態：最多 3 則

審核流程實作（Sprint 11）：
1. 用戶申請 → 系統產生 6 碼隨機碼（英數，10 分鐘有效）
2. 用戶上傳含隨機碼的自拍照 → status=pending
3. 後台管理員於 /admin/verifications 審核照片
4. 核准 → membership_level=1.5，credit_score+15，Email 通知
5. 拒絕 → 附原因，Email 通知，用戶可重新申請

進階驗證男會員 (Level 2) - 男性「信用卡付款方式可用」即取得
│  條件（任一即可）：
│  - 完成 NT$100 信用卡驗證（付款後可退還）
│  - 曾有任何 ECPay 付款成功（訂閱方案 / 體驗方案 / 點數購買）
│  - Admin 後台手動授予（fallback 用，子系統故障時使用）
│  業務語意：Lv2 = 信用卡付款方式可用，不論透過驗證或實際付款證明。
│  預設功能（可後台調整，累加 Level 1）：
├─ 查看對方完整個人資料
└─ 發送動態：最多 1 則

付費會員 (Level 3) - 購買任一訂閱方案後取得
│  升級條件：必須先完成進階驗證（Level 1.5 或 Level 2）
│  預設功能（可後台調整，累加所有前置等級）：
├─ 無限聊天
├─ 已讀回執
├─ VIP 特權：隱身模式、廣播訊息
└─ 搜尋結果優先顯示
```

**實作對應（各等級的 ground truth 欄位）**：

| 等級 | 升級條件（規格原文） | 程式碼判定欄位 | 主要寫入路徑 |
|---|---|---|---|
| Lv0 | 註冊完成 Email 驗證 | `users.email_verified=true` | 註冊流程 |
| Lv1 | 手機 SMS 驗證 | `users.phone_verified=true` | `AuthController::verifyPhone` |
| Lv1.5 | 女性完成真人照片驗證 | `user_verifications.status='approved'`（gender=female）| `Admin/V1/VerificationController::review` |
| Lv2 | 男性信用卡付款方式可用（NT$100 驗證 OR 任一 ECPay 付款 OR admin 升） | `gender='male'` 且（`users.credit_card_verified_at IS NOT NULL` OR exists `payments.paid_at IS NOT NULL`） | `VerificationHandler::onPaid` / 任一 ECPay 付款 callback / `AdminController::moderateUser` action=verify_advanced |
| Lv3 | 購買任一訂閱方案 | `subscriptions.status='active' AND expires_at > now()` | `PaymentService::activateSubscription` |

訂閱到期降級邏輯（`subscriptions:expire` job）依此表反推 base level：
- 男性已完成信用卡驗證者降回 Lv2，未完成者降回 Lv1
- 女性照片驗證已通過者降回 Lv1.5，未通過者降回 Lv1
- 已驗手機但未做進階驗證者降回 Lv1，未驗手機者降回 Lv0
- 詳見 `User::getBaseMembershipLevel()` 與 DEV-005 §10.4

> **Admin 手動升降級的 edge case**（admin `verify_advanced` / `unverify_advanced` 與 ground truth 欄位不對稱）已記錄於 `docs/decisions/2026-05-03-admin-membership-edge-cases.md`，待後續專門 sprint 處理。

### 4.2 MVP功能規格 (第一版必須功能)

#### 4.2.1 用戶註冊與驗證系統

**功能描述**：提供安全、便捷的用戶註冊流程  
//先產生帳號再後補個人資料

**功能規格**：
- **註冊流程**：
  - 輸入基本資料（性別、暱稱、生日、Email、密碼）
  - 強制閱讀並同意條款（隱私權政策、使用條款、防詐指南）
  - Email 驗證（發送驗證碼）

- **個人資料完善**：
  - **個人資料**：年齡、身高、體重、地區、菸酒情況、自我介紹；選填：教育程度、工作、收入、身體特徵、常見嗜好（提供清單勾選）
  - **上傳照片**：大頭貼 1 張 + 附加照片最多 2 張
  - **填寫預算**：
    - 單次：8K 以下 / 8～12K / 12K 以上
    - 長期：40K 以下 / 40～60K / 60K 以上
  - **篩選偏好**：可設定願意接受的對方誠信等級下限

- **身份驗證**：
  - **手機號碼驗證**（SMS 驗證碼） → 完成後升為「驗證會員 Lv1」
  - **女性進階驗證**：上傳包含隨機驗證碼的自拍照，由管理員審核；**自拍照須永久保留於雲端**，以備爭議查核 → 審核通過後升為「驗證女會員 Lv1.5」
  - **男性進階驗證**：信用卡驗證（收費 NT$100，串接綠界）→ 付款成功後升為「進階驗證男會員 Lv2」
  - **驗證碼系統**：隨機 6 位數字。**Email 驗證碼 10 分鐘有效**；**手機 SMS OTP 5 分鐘有效**（業界慣例 + 簡訊成本控制）；**女性進階驗證隨機碼 10 分鐘有效**

**驗收標準**：
```
Given 用戶訪問註冊頁面
When 用戶填寫完整註冊資訊並提交
Then 系統發送 Email 驗證碼
And 用戶完成 Email 驗證後進入手機驗證
And 完成手機驗證後成為「驗證會員 Lv1」

Given 女性驗證會員申請進階驗證
When 系統顯示隨機 6 位驗證碼（10 分鐘有效）
And 用戶上傳手持驗證碼自拍照
Then 照片上傳至後台待審核，並永久保留於雲端
And 管理員審核通過後用戶升為「驗證女會員 Lv1.5」，誠信分數 +15

Given 男性驗證會員申請進階驗證
When 用戶完成信用卡付款 NT$100（綠界）
Then 系統確認付款成功，用戶升為「進階驗證男會員 Lv2」，誠信分數 +15
```

#### 4.2.2 誠信分數系統

**功能描述**：創新的用戶信用評價系統，建立平台信任機制

> 評分依據：以**發生在平台上的行為紀錄**評估用戶可信度。對外（前台）**僅顯示用戶所屬等級分類**（頂級／優質／普通／受限），**暫不顯示具體分數數值**，以降低用戶對數值的焦慮感，並保留後台調整彈性。


**分數規則**：
- **初始分數**：註冊用戶 **60 分**（DEV-008 §3；後台可調整 `credit_score_initial`）
- **分數區間**：0-100 分，分為 4 個等級
  - 0 分：停權
  - 1-30 分：受限用戶
  - 31-60 分：普通用戶
  - 61-90 分：優質用戶
  - 91-100 分：頂級用戶

> 分數上限 100 分，所有加減分值儲存於 `system_settings`（後台可即時調整），
> 詳細 key 清單見 DEV-008 §4 / §5。

**加分機制**（預設值，後台可調）：
- 完成 Email 驗證：+5 分（`credit_add_email_verify`）
- 完成手機驗證：+5 分（`credit_add_phone_verify`）
- 完成進階驗證（男性信用卡）：+15 分（`credit_add_adv_verify_male`）
- 完成進階驗證（女性照片）：+15 分（`credit_add_adv_verify_female`）
- 約會驗證成功（GPS 通過）：+5 分（`credit_add_date_gps`）
- 約會驗證成功（無 GPS）：+2 分（`credit_add_date_no_gps`）
- 管理員獎勵：可變分數

**扣分機制**：
- 約會爽約（後台認定）：-10 分（`credit_sub_date_noshow`）
- 提交或收到檢舉：-10 分（`credit_sub_report_user`）
- 發布不當內容（管理員刪除）：-5 分（`credit_sub_bad_content`）
- 惡意騷擾（管理員認定）：-20 分（`credit_sub_harassment`）
- 檢舉屬實額外處分：-5 分（`credit_sub_additional_penalty`）
- 管理員懲罰：可變分數（-1 至 -50）

**權限控制**：
- 高分數區間用戶可向低分數用戶發送訊息
- 低分數用戶無法主動聯繫高分數用戶
- 30分以下用戶功能受限

<font color=red>因觸發條件眾多須測試所有情況是否正常運作</font>

**驗收標準**：
```
Given 用戶誠信分數為85分
When 用戶成功完成約會驗證
Then 誠信分數增加5分變為90分
And 系統記錄分數變更日誌
And 用戶收到分數變更通知
```

#### 4.2.3 QR碼約會驗證系統

**功能描述**：革命性的約會真實性驗證機制，提升雙方互信

**功能流程**：
1. **約會邀請**：
   - 用戶A向用戶B發送約會邀請
   - 設定約會時間、見面地點
   - 系統生成唯一QR碼
   
   > **約會邀請入口（v1.3 更新）：**
   > - 入口一（原有）：聊天對話頁 → 點擊 QR 碼 icon → 發起邀請
   > - 入口二（新增）：個人資料頁 → 「📅 邀請約會」按鈕 → 自動建立對話 → Bottom Sheet 填寫
   > - 入口二顯示條件：邀請方 Lv3（付費）+ 被邀請方 Lv1+（驗證）
   > - 兩個入口共用相同的後端 API（POST /api/v1/dates）

2. **QR碼特性**：
   - 時間限定：約會時間前後30分鐘內有效
   - 位置限定：GPS範圍500公尺內有效
   - 一次性使用：掃描後立即失效

3. **驗證流程**（SPEC-CONFIRM-001 B.6 更新）：
   - 雙方在約定時間地點掃描對方QR碼
   - 系統驗證時間、身份，GPS 位置驗證為**選擇性**
   - **驗證成功（GPS 通過）**：雙方各獲得 +5 分誠信分數
   - **驗證成功（GPS 未提供/失敗）**：雙方各獲得 +2 分誠信分數
   - 得分值由後台 `system_settings` 可動態調整


**技術規格**：
- QR碼內容：加密的約會ID + 時間戳 + 用戶ID
- 有效期：約會時間前後各30分鐘
- GPS驗證：選擇性，通過標準為 ±500 公尺範圍內（Haversine 公式計算）
- 前端 GPS 流程（v1.4 含授權提示畫面）：
  1. 掃碼成功 → 顯示「GPS 授權說明」畫面（說明 +5 vs +2 差異）
  2. 用戶點「允許定位並驗證」→ 此時才觸發 iOS/Android 系統授權彈窗
  3. 用戶點「跳過定位，直接驗證」→ 不觸發系統授權，直接送出
  4. GPS 取得成功：連同座標送至後端（可獲 +5 分）
  5. GPS 授權拒絕或逾時（10 秒）：仍送出驗證，不附座標（獲 +2 分）
  6. 前端顯示 GPS 狀態：「📍 GPS 驗證通過」或「📍 GPS 未通過」
  7. 設計理由：iOS/Android 要求 GPS 前需提供使用目的說明，
     先顯示自訂說明畫面再觸發系統彈窗，提高用戶同意率
- 防作弊：同一約會只能驗證一次（token 掃碼後立即失效）

**驗收標準1約會成功**：
```
Given 用戶A邀請用戶B約會，設定時間為19:00
When 雙方在18:45-19:30之間成功掃描對方QR碼
And 雙方提供 GPS 且均在約定地點 500 公尺內
Then 系統確認約會成功，雙方誠信分數各增加 5 分

When 雙方成功掃描QR碼但未提供 GPS 或 GPS 超出範圍
Then 系統仍確認約會成功，雙方誠信分數各增加 2 分
```
**驗收標準2約會爽約**：
```
Given 用戶A邀請用戶B約會，設定時間為19:00
When 超過約定時間，用戶B未出現
And 於現場用戶A回報爽約並提供 GPS 且在約定地點 500 公尺內
Then 系統於19:30確認約會爽約，用戶B誠信分數扣除10分

#### 4.3.1 智能搜尋與配對系統

**功能描述**：基於多維度算法的智能配對推薦系統

**搜尋功能**：

```

1. 誠信分數（降序）
2. 最後上線時間（降序）
3. 配對分數（降序）
4. 註冊時間（升序）

**驗收標準**：
```

#### 4.4.4 誠信分數自動停權機制

**功能描述**：當用戶誠信分數歸零時，系統自動觸發帳號停權，無需等待管理員手動操作

**觸發條件**：用戶誠信分數降至 0 分（含）以下

**停權後行為**：
- 用戶登入後僅能訪問「個人資料頁」及「申訴頁面」
- 無法瀏覽其他用戶、發送訊息、查看動態
- 搜尋結果中不出現此帳號
- 系統發送 Email 通知用戶帳號已被暫停，並說明申訴流程

**申訴流程**：
- 用戶在申訴頁面填寫申訴理由（可附圖）
- 後台管理員審核申訴，可選擇：補回分數並解除停權 / 維持停權 / 手動刪除帳號

**分數補回與解停**：管理員審核通過後手動調整分數至 30 分以上，系統自動解除停權限制

**驗收標準**：
```

#### 4.4.6 申訴流程完整規格（Sprint 8 補完）

**功能描述**：停權用戶的完整申訴與解停閉環

**前台申訴流程（用戶端）**：

申訴入口：停權後自動跳轉 `/suspended`，頁面提供：
- 停權原因（來自最後一次觸發停權的 credit_score_history.reason）
- 目前誠信分數（0 分）
- 「我要申訴」按鈕

申訴頁（`/suspended/appeal`）：
- 必填：申訴說明（最多 500 字）
- 選填：上傳佐證截圖（最多 3 張，每張 5MB 以內）
- 提交後：顯示案號（格式 A + 年份 + 5 位流水號，如 A202600001）
- 申訴送出後顯示：「申訴已送出，我們將在 3 個工作天內回覆，請保持 Email 通暢。」

**申訴頻率限制（PR-C 拍板）**：
- 同停權期間最多 **3 次** 申訴（不論已處理 / 審核中均計入）
- 同時最多 **1 筆** 審核中（pending / investigating）
- 「停權期間」起算 = `users.suspended_at`（user 最近一次被停權的時間）
- 達到 3 次後拒絕新申訴 → 422 + `code='APPEAL_LIMIT_REACHED'`
- 已有審核中申訴時送出新申訴 → 422 + `code='APPEAL_EXISTS'`
- user 解停後若再次觸發停權，計數從新的 `suspended_at` 起算（自然歸零）

**後台審核流程（管理員端，D.3 解耦版 — 2026-05-02 起）**：

申訴列表入口：後台 Ticket 列表 → 類型篩選「申訴」（type = appeal）

申訴 Ticket 詳情頁顯示（GET `/admin/tickets/{id}`）：
- 用戶基本資料（暱稱、Email、停權前誠信分數）
- 申訴說明與佐證截圖（最多 3 張）
- 該用戶完整誠信分數歷史（最近 20 筆）
- 該用戶被檢舉記錄（最近 20 筆，排除本筆自身）
- 醒目提示：「處理此申訴 ticket 僅變更 ticket 狀態，不影響使用者帳號狀態。如需解停使用者，請至『會員管理』頁面操作。」

**Ticket 系統與 user.status / 誠信分數完全解耦：**
管理員處理申訴 ticket 只能變更 ticket.status（pending → investigating → resolved/dismissed）+ 填寫 admin_reply。
**不再**連動 user.status 或 credit_score。

管理員操作選項（兩種）：
① **核准申訴**（status = resolved）
  - 輸入給用戶的回覆說明（必填）
  - 執行後：ticket.status = resolved + 系統依 user.status 自動發送通知（站內訊息 / email）
  - **使用者帳號狀態不會自動恢復**，需 admin 至「會員管理 → 解停」獨立操作
② **駁回申訴**（status = dismissed）
  - 輸入駁回原因（必填）
  - 執行後：ticket.status = dismissed + 系統依 user.status 自動發送通知

**解停 / 補分流程（與 ticket 完全獨立）：**
- 解停 user：admin 後台 → 會員管理 → 該帳號 → 「解除停權」按鈕（PATCH /admin/members/{id}/actions, action='unsuspend'）
- 補誠信分數：admin 後台 → 會員管理 → 該帳號 → 「調整分數」按鈕（PATCH /admin/members/{id}/actions, action='adjust_score'）

**自動解停的另一條觸發（保留既有 Observer 邏輯）：**
- `credit_score >= 30` AND `status = 'auto_suspended'` 時 CreditScoreObserver 自動解停
- 此觸發**不依賴 ticket 處理**，由分數變動本身觸發（admin 手動補分時也會觸發）
- 解停後 status → 'active'，清除 Cache `suspended_user:{id}`
- 發送 AccountReactivatedMail

---

**Ticket 處理通知雙軌（D.3 新增 — 2026-05-02）：**

當 ticket.status 變為 resolved/dismissed 時，系統依「處理當下」`reporter.status` 自動發送通知：

| 情境 | reporter.status | 通知管道 | 通知類型 |
|---|---|---|---|
| 1 | active | 站內訊息 | `notifications` 表 type=`ticket_replied` |
| 2 | suspended / auto_suspended | Email | `TicketProcessedMail`（Resend，queue 派發）|

**Email 範本三變體：**
- type=appeal + status=resolved：主旨「您的申訴處理結果通知」+ **明確說明「申訴核准 ≠ 帳號自動恢復」**+ 引導聯繫客服
- type=appeal + status=dismissed：主旨「您的申訴未通過」+ 駁回理由
- type≠appeal：主旨「您的回報已處理」+ admin_reply

**重複防護：** 重複設定同樣終態（如 resolved → resolved）**不重複發送**通知。

```
驗收標準：
Given 用戶誠信分數歸零被自動停權
When 用戶登入
Then 自動跳轉 /suspended 頁面，顯示停權原因
When 用戶填寫申訴並送出（含說明文字）
Then 建立申訴 Ticket（type=appeal），顯示案號 A202600001
And 同一停權期間再次提交回傳 422
When 管理員在後台選擇「核准申訴」並將分數調整至 30 分以上
Then 用戶分數更新，系統自動解除停權
And 系統自動將 status 改為 'active'
And 發送解停 Email 給用戶
And 用戶刷新頁面後恢復正常功能
When 管理員選擇「駁回申訴」
Then 維持停權狀態
And 發送駁回 Email 給用戶（含駁回原因）
```

#### 4.4.7 隱私設定規格（Sprint 8 補完）

#### 4.4.8 帳號刪除 GDPR 完整規格（Sprint 8 補完）

**功能描述**：符合 GDPR 的用戶資料刪除權，含冷靜期與資料匿名化

**前台申請流程**：

入口：`/app/settings` → 「帳號設定」→「刪除帳號」

三步驟流程：
1. **確認頁**：顯示「刪除後的影響」清單
   - 所有對話紀錄將在 30 天後永久刪除
   - 誠信分數與驗證資料無法回復
   - 若有有效訂閱，剩餘天數不退款
   - 女性進階驗證照片依法規保留 1 年

2. **密碼驗證**：重新輸入密碼確認（防誤觸）

3. **送出申請**：
   - 帳號進入「待刪除冷靜期」（7 天）
   - 冷靜期間帳號仍可正常登入，並顯示橙色倒數 Banner
   - 顯示：「您的帳號將於 {date} 永久刪除。若要取消，請在到期前至設定頁操作。」
   - 冷靜期間可隨時取消刪除申請

**刪除執行流程（後端排程）**：

每日 03:00 排程執行：
- 查詢 `users.status = 'pending_deletion'` 且 `delete_requested_at < NOW() - 7 days`
- 執行資料匿名化：
  - `email` → `deleted_{id}@removed.mimeet`
  - `phone` → NULL
  - `nickname` → `已刪除用戶`
  - `avatar_url` → NULL（圖片從 CDN 刪除）
  - `profile` JSON → NULL
  - `privacy_settings` → NULL
  - `password` → 隨機雜湊（使帳號無法登入）
  - `deleted_at` → NOW()
  - `status` → 'deleted'
- **保留**：users.id、uuid（供歷史 FK 完整性）、gender（統計用）
- **保留 1 年**：`user_verifications.verification_file_url`（女性照片，合規要求）
- 發送「帳號已刪除」確認 Email（在匿名化前取得 email）
- 記錄到 admin_operation_logs（type: gdpr_deletion）

**取消刪除**：
- 冷靜期間用戶可在設定頁點擊「取消刪除申請」
- status 改回 'active'，`delete_requested_at` 清空

```
驗收標準：
Given 用戶通過密碼驗證並送出刪除申請
Then status → 'pending_deletion'，delete_requested_at = NOW()
And 用戶看到 7 天倒數 Banner
And Email 收到「帳號刪除申請已受理」確認信
Given 用戶在冷靜期間點擊「取消刪除申請」
Then status → 'active'，倒數 Banner 消失
Given 7 天冷靜期結束（排程觸發）
Then email, phone, nickname 等欄位匿名化
And CDN 圖片刪除（照片驗證圖片保留 1 年）
And 發送「帳號已永久刪除」Email
And admin_operation_logs 記錄 gdpr_deletion
```
