## 跨功能觀察

### 1. Phase 2 超前實作密度高

原規格將 31 項放 Phase 2，目前已完成 22 項（71%）。**剩餘 9 項中有 5 項是社群動態系統（F29~F32 + F35 + F44 + A11），互相依賴成一塊**。社群動態子系統屬於可以一次切片開發的完整模組，建議當作獨立 sprint 處理。

### 2. 孤兒型不一致

兩項出現「一棧做了另一棧沒做」：
- **A18**：前端 router 與 view 已建，後端 commented stubs → 前端會 404
- **A02**：後端 endpoint 已建，前端不曾呼叫 → 後端 dead code

兩者風險不同：A18 影響用戶體驗（404 頁面），A02 只是 dead code（未掛入儀表板）。

### 3. IMPLEMENTATION_STATUS.md 已部分過時

2026-04-20 版本宣告 100% MVP 完成正確，但 Phase 2 描述有以下與現況不符：
- 文中說「A20 角色權限管理 → ✅ AdminUsersPage」— 實際上是 `SystemSettingsPage.tsx` 內嵌的管理員 section（無獨立 `AdminUsersPage.tsx` 檔案）
- 文中說「A18 ⏳ 保留 Phase 2」未提及前端路由與 view 已先行存在（孤兒狀態）

審計時優先看 code 不看 doc 是正確的（CLAUDE.md 在 §1 警告過此事）。

### 4. F28 智能排序符合 spec 降階版本

spec 本身建議「Phase 1 先用『誠信分數 + 最後上線時間』排序」，目前 `UserController::search` 排序 = 資料完整度 → credit_score → last_active_at。雖然 spec 的完整算法（偏好匹配 + 照片吸引力）未實作，但這是規格本身允許的降階版本，**不算缺漏**。

### 5. Ticket / Report 統一設計

F43、F45、F46、F15、A13 全部走同一個 `reports` 表 + Ticket 系統，type 欄位區分 `report / system / appeal / unsubscribe / anon_report`。此設計簡化後台 ops，但代表 `anon_report` 的接收端在 F35 上線前都是死值。
