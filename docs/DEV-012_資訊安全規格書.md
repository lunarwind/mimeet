## 2. 詳細弱點清單

### VULN-001：Broadcast Channel 認證繞過（Critical）

- **風險等級：** Critical
- **CVSS 評分：** 9.1
- **位置：** `backend/routes/channels.php:18`
- **說明：** `chat.{conversationId}` channel 的授權回調固定回傳 `true`，任何已認證用戶皆可監聽任意聊天頻道，導致私訊內容可被第三方竊聽。
- **PoC：** 以用戶 A 登入 → 訂閱 `private-chat.{用戶B的conversationId}` → 可收到用戶 B 與 C 的所有即時訊息。
- **修復方案：** 回調中驗證 `$user` 是否為該 `$conversationId` 的參與者（查詢 `conversations` 表）。
- **對應 Task：** S13-01, S13-12

### VULN-002：後台登入端點缺少 Rate Limiting（Critical）

- **風險等級：** Critical
- **CVSS 評分：** 8.8
- **位置：** `backend/routes/api.php:164`
- **說明：** `POST /api/v1/admin/auth/login` 無 throttle middleware，攻擊者可無限次嘗試密碼暴力破解。前台登入已有 `throttle:5,1` 保護，但後台完全未設限。
- **PoC：** 對 `/api/v1/admin/auth/login` 發送 1000 次不同密碼請求，全部正常回應 401，無任何速率限制。
- **修復方案：** 加入 `->middleware('throttle:3,1')` 並實作連續失敗 5 次鎖定 15 分鐘機制。
- **對應 Task：** S13-02, S13-09

### VULN-003：Sanctum Token 永不過期（High）

- **風險等級：** High
- **CVSS 評分：** 7.5
- **位置：** `backend/config/sanctum.php:29`
- **說明：** `expiration` 設為 `null`，Token 一經核發即永久有效（除非手動撤銷）。若 Token 洩漏，攻擊者可無限期使用。後台管理員 Token 同樣受影響。
- **修復方案：** 前台設定 `expiration => 1440`（24 小時），後台 Token 另建 guard 設定 `480`（8 小時）。
- **對應 Task：** S13-08

### VULN-004：缺少 HTTP Security Headers（High）

- **風險等級：** High
- **CVSS 評分：** 7.2
- **位置：** `docker/nginx/default.conf`（整個檔案）
- **說明：** Nginx 配置完全缺少安全標頭：無 HSTS、無 CSP、無 X-Frame-Options、無 X-Content-Type-Options、無 Referrer-Policy。且 `server_tokens` 未關閉，洩漏 Nginx 版本。
- **修復方案：** 在 Nginx server block 中加入完整安全標頭（見 §3.1）。
- **對應 Task：** S13-03

### VULN-005：Phone 欄位明文儲存（High）

- **風險等級：** High
- **CVSS 評分：** 7.0
- **位置：** `backend/app/Models/User.php:40, 67-81`
- **說明：** `phone` 欄位在 `$fillable` 中但未使用加密 cast，以明文儲存於資料庫。DEV-001 規格書要求使用 AES-256-CBC 加密，但實際未實作。若資料庫洩漏，所有用戶手機號碼將直接暴露。
- **修復方案：** 在 `$casts` 中加入 `'phone' => 'encrypted'`。
- **對應 Task：** S13-06

### VULN-006：User Model $fillable 含管理員欄位（Medium）

- **風險等級：** Medium
- **CVSS 評分：** 6.5
- **位置：** `backend/app/Models/User.php:24-50`
- **說明：** `$fillable` 包含 `credit_score`、`status`、`membership_level` 等管理員專用欄位。雖然 Controllers 目前使用 `$request->only([...])` 限制，但若有開發者疏忽使用 `$request->all()` 即可造成權限提升。
- **修復方案：** 將管理員欄位從 `$fillable` 移除，改用專用方法更新。
- **對應 Task：** S13-14

### VULN-007：上傳僅驗證 MIME/Extension，未驗證 Magic Bytes（Medium）

- **風險等級：** Medium
- **CVSS 評分：** 5.8
- **位置：** `backend/app/Http/Controllers/Api/V1/UserController.php:110`、`ReportController.php:26`
- **說明：** 檔案上傳使用 `mimes:jpeg,png,gif,webp` 驗證，僅檢查副檔名和 MIME header。攻擊者可偽造 MIME type 上傳 PHP shell 或惡意腳本。
- **修復方案：** 加入 `getimagesize()` 或 `finfo_file()` 進行 magic bytes 驗證。
- **對應 Task：** S13-10

### VULN-008：Admin Token 無 IP 綁定（Medium）— ✅ 已修復

- **風險等級：** Medium
- **CVSS 評分：** 5.5
- **位置：** `backend/app/Http/Controllers/Api/V1/AdminController.php`
- **說明：** 後台管理員 Token 核發後不綁定 IP，被竊取的 Token 可在任意 IP 使用。
- **修復方案：** 登入時將 IP 嵌入 token name（`admin-token-{ip}`），`EnsureAdminUser` middleware 每次請求解析 token name 比對當前 IP，不符時返回 401 並寫 Log::warning。
- **修復實作：**
  - `AdminController::login()` — token name 改為 `admin-token-{$loginIp}`，同步更新 `admin_users.last_login_ip`
  - `EnsureAdminUser::handle()` — 新增 IP binding check，`str_starts_with('admin-token-')` 判斷後比對 IP
  - Migration `2026_04_23_090045` — `admin_users` 新增 `last_login_ip VARCHAR(45) NULLABLE`
  - 相容：舊 token（name=`admin-token`，無 IP 後綴）自動放行，下次登入後綁定新 IP
- **對應 Task：** S13-08
- **修復 Commit：** security(admin): bind admin token to login IP on issue, verify on each request (VULN-008)

### VULN-009：前端 console.log 殘留（Medium）

- **風險等級：** Medium
- **CVSS 評分：** 3.5
- **位置：** 13 處（usePayment.ts:6處、useProfile.ts:2處、useInfiniteScroll.ts:2處、useImageUpload.ts:1處、QRScanView.vue:1處、ReportsHistoryView.vue:1處）
- **說明：** 前端殘留 13 處 console.error/console.warn 呼叫。雖未洩漏敏感資料，但可能在 production 環境向用戶暴露內部錯誤訊息。
- **修復方案：** 移除所有 console.log/error/warn，改用 error tracking 服務（如 Sentry）。
- **對應 Task：** S13-07

### VULN-010：ECPay Webhook 端點接受 GET（Low）

- **風險等級：** Low
- **CVSS 評分：** 3.1
- **位置：** `backend/routes/api.php:118`
- **說明：** ECPay return URL 使用 GET 方法，雖為用戶瀏覽器跳轉用途，但 `/mock` 端點在非 local 環境應移除。
- **修復方案：** 確保 `/mock` 端點僅在 local 環境可用。
- **對應 Task：** 追蹤中

### VULN-011：CORS 僅限 localhost（Low）

- **風險等級：** Low
- **CVSS 評分：** 2.0
- **位置：** `backend/config/cors.php`
- **說明：** `allowed_origins` 僅含 `localhost:5173` 和 `localhost:5174`。部署至生產環境前需更新為實際域名，否則前端無法正常運作。此為部署配置項目，非安全弱點。
- **修復方案：** 部署前更新 `allowed_origins` 為生產域名。
- **對應 Task：** 追蹤中

## 3. 安全基線規格

### 3.1 必要 HTTP Security Headers

```nginx
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
add_header Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src 'self' https://fonts.gstatic.com; img-src 'self' data: https://cdn.mimeet.tw; connect-src 'self' wss://api.mimeet.tw" always;
add_header X-Frame-Options "DENY" always;
add_header X-Content-Type-Options "nosniff" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Permissions-Policy "camera=(self), microphone=(), geolocation=(self)" always;
server_tokens off;
```

### 3.2 Rate Limiting 規格

| 端點 | 限制 | 鎖定時間 |
|------|------|---------|
| POST /auth/login | 5次/15分鐘（同IP） | 15分鐘 |
| POST /auth/register | 3次/小時（同IP） | 1小時 |
| POST /auth/forgot-password | 3次/小時（同IP） | 1小時 |
| POST /auth/verify-phone/send | 3次/小時（同用戶） | 1小時 |
| POST /uploads | 10次/分鐘（同用戶） | - |
| GET /users/search | 20次/分鐘（同用戶） | - |
| POST /dates/verify | 1次有效/24小時（同對） | - |
| 後台 POST /admin/auth/login | 5次/15分鐘（同IP） | 15分鐘 |

### 3.3 敏感欄位規格

| 欄位 | 儲存方式 | API 輸出規則 |
|------|---------|------------|
| password | bcrypt (hashed cast) | 永不輸出（$hidden） |
| phone | AES-256-CBC (encrypted cast) | 遮罩（09xx-xxx-xxx）|
| credit_score | 明文 | 僅輸出等級分類給前台 |
| email | 明文 | 僅輸出給本人 |
| ip_address（日誌）| 明文 | 需 show_ip=true + super_admin |

### 3.4 檔案上傳規格

- **允許格式：** jpg / jpeg / png / gif / webp（magic bytes 驗證）
- **最大大小：** 5MB
- **儲存位置：** DO Spaces（非 public/ 目錄）
- **存取方式：** Signed URL（有效期 1 小時）
- **AI 審核：** Sightengine（avatar / profile_photo / post_image）

### 3.5 後台 Token 規格

- **有效期：** 8 小時（28800 秒）
- **IP 綁定：** 登入時記錄 IP，每次請求驗證 IP 一致性
- **失效條件：** 登出 / 到期 / IP 變更 / 管理員停用

## 4. 安全開發規範（Secure Coding Guidelines）

### 4.1 後端（Laravel）

```php
// ✅ 正確：使用 Eloquent 參數綁定
User::where('email', $email)->first();

// ❌ 錯誤：原始 SQL 拼接
DB::select("SELECT * FROM users WHERE email = '$email'");

// ✅ 正確：使用 Policy 驗證 ownership
$this->authorize('view', $conversation);

// ❌ 錯誤：直接 findOrFail 無 ownership 驗證
$conv = Conversation::findOrFail($id);

// ✅ 正確：明確 fillable（不含管理欄位）
protected $fillable = ['nickname', 'bio', 'city'];

// ❌ 錯誤：允許 mass assignment 管理欄位
protected $fillable = ['nickname', 'bio', 'credit_score', 'status'];

// ✅ 正確：Broadcast Channel 驗證參與者
Broadcast::channel('chat.{conversationId}', function ($user, $conversationId) {
    return Conversation::where('id', $conversationId)
        ->where(fn($q) => $q->where('user_a_id', $user->id)->orWhere('user_b_id', $user->id))
        ->exists();
});

// ❌ 錯誤：所有人都能監聽
Broadcast::channel('chat.{conversationId}', function ($user, $conversationId) {
    return true;
});
```

### 4.2 前端（Vue.js）

```vue
<!-- ✅ 正確：Mustache 語法自動 escape -->
<span>{{ userInput }}</span>

<!-- ❌ 錯誤：v-html 直接渲染用戶輸入 -->
<span v-html="userInput"></span>

<!-- ✅ 正確：若需渲染 HTML，先 sanitize -->
<span v-html="sanitizeHtml(userInput)"></span>
```

### 4.3 API 設計原則

1. 所有 `/app/*` 路由必須有 `auth:sanctum` middleware
2. 所有資源存取必須驗證 ownership（使用 Policy）
3. 所有 POST/PATCH/DELETE 必須有 Rate Limiting
4. Response 不得含 stack trace（`APP_DEBUG=false` in production）
5. 後台 API 必須有 `admin.log` middleware 記錄操作
