## 1. 核心設計原則

| 原則 | 說明 |
|------|------|
| **起始信任** | 新用戶預設高分（60分），違規才扣分，而非從零累積 |
| **雙向懲罰** | 檢舉時雙方同步扣分，防止惡意濫用檢舉 |
| **自動閉環** | 分數歸零自動停權，免除管理員手動介入延遲 |
| **可動態調整** | 所有加減分值存於 `system_settings`，後台可即時修改，無需改程式碼 |
| **完整稽核** | 每次分數異動都寫入 `credit_score_logs`，可追溯原因與操作者 |

> **設計原則：規格 = Code Default**
> 本規格書中所有分數值（含 §3.2 加分、§3.3 扣分、§3.4 管理員裁量）即為 Service 層
> `SystemSetting::get($key, $default)` 中的 `$default` 參數值。
> - 系統初始化時，`SystemSettingsSeeder` 將這些值寫入 `system_settings` 表
> - 後台「系統設定 → 誠信分數配分」可即時調整 DB 值（儲存後快取失效，下一筆觸發即生效）
> - 若 DB row 缺失或毀損，Service 讀取自動 fallback 至本規格書的預設值
> - **扣分 key 一律存正值**（`credit_sub_*` = 正整數），Service 內統一轉負（`-getConfig(...)`）
> - 後台管理員獎勵範圍：`credit_admin_reward_min` ~ `credit_admin_reward_max`（預設 1~20）
> - 後台管理員懲罰範圍：`credit_admin_penalty_min` ~ `credit_admin_penalty_max`（預設 1~20，對稱）

## 2. 分數範圍與等級定義

```
分數範圍：0 ~ 100（整數）
邊界行為：
  - 加分超過 100 → 上限鎖定 100，不溢出
  - 扣分低於 0   → 下限鎖定 0，同時觸發自動停權
```

| 等級 | 分數區間 | 標籤 | 顏色 token | 核心限制 |
|------|---------|------|-----------|---------|
| 頂級 | 91 – 100 | 頂級 | `credit-top` | 無 |
| 優質 | 61 – 90  | 優質 | `credit-good` | 無 |
| 普通 | 31 – 60  | 普通 | `credit-normal` | 無法向高分數用戶發起訊息（見 §4） |
| 受限 | 1 – 30   | 受限 | `credit-low` | 同上，且排序後置 |
| 停權 | 0        | 停權 | `credit-suspended` | 自動停權，僅能存取個人頁 + 申訴頁 |

### 5.3 檢舉結案補回機制

```
檢舉結案流程：
  管理員審核後選擇「屬實」
    → 被檢舉人：維持扣分（-10 或 -5 不補回）
    → 檢舉人：補回原扣分值（+10 或 +5）
    → 管理員可額外懲罰被檢舉人（另行手動扣分）

  管理員審核後選擇「不屬實」
    → 被檢舉人：補回原扣分值
    → 檢舉人：不補回（視為惡意檢舉成本）
    → 【待確認】是否對惡意檢舉人額外再扣分？
```

## 7. 自動停權機制

### 7.1 觸發條件

```
當 users.credit_score 被更新後：
  if (新分數 ≤ 0) AND (原狀態 = 'active' OR 'restricted')
    → 執行自動停權
```

> **精確邏輯**：`credit_score <= 0`（含等於0，不是 <0）

### 7.2 停權後行為

```
users.status 更新為 'auto_suspended'
Cache::put("suspended_user:{user_id}", true)

允許存取的路由白名單：
  GET  /app/profile/me       (個人資料頁，唯讀)
  GET  /app/appeal           (申訴頁入口)
  POST /api/v1/appeals       (提交申訴)
  POST /api/v1/auth/logout   (登出)

禁止存取（回傳 403 + 停權提示）：
  所有其他前台功能頁面與 API

前台表現：
  - 進入 App 後自動導向 /suspended 頁面
  - 頁面顯示：停權原因說明 + 申訴入口
  - Email 通知：AccountAutoSuspendedMail
```

### 7.3 解停流程

```
用戶提交申訴 → 管理員審核

  路徑 A（核准申訴）：
    管理員手動將 credit_score 調整至 ≥ 30
    → Observer 偵測 credit_score 恢復且 status = 'auto_suspended'
    → 自動更新 status = 'active'
    → 清除 Cache "suspended_user:{id}"
    → 發送 AccountReactivatedMail

  路徑 B（駁回申訴）：
    維持停權，管理員可附上說明（寫入 ticket 回覆）

  路徑 C（管理員直接刪除帳號）：
    users.deleted_at 軟刪除（僅 super_admin 可操作）
```

### 7.4 解停臨界分數

> **【已確認 - Sprint 8】** 解停臨界分數為 `credit_score ≥ 30`。
> 管理員核准申訴時，補回分數最低必須設定 30 分（系統強制驗證）。
> 理由：30 分為「普通」等級下緣，確保解停後用戶具備基本使用資格。

### 7.5 申訴 Ticket 自動建立邏輯

用戶提交 `POST /api/v1/me/appeal` 時：

1. 驗證 `users.status = 'auto_suspended'`（否則 422 NOT_SUSPENDED）
2. 查詢同一停權期間是否已有 appeal Ticket（否則 422 APPEAL_EXISTS）
   - 查詢條件：`reports.type='appeal' AND reports.reporter_id=userId AND reports.created_at > 本次停權時間`
3. 建立 reports 記錄：
   - `type = 'appeal'`
   - `reporter_id = userId`
   - `reported_user_id = userId`（自己申訴自己的停權）
   - `description = 用戶申訴說明`
   - `status = 'pending'`
4. 上傳佐證截圖至 Storage
5. 觸發後台通知（站內通知給所有 admin）

## 8. 訊息發送權限控制（誠信區間規則）

依 xlsx 規格：「較高區間可向低區間發送訊息，反之不可發送訊息」

```
規則定義：
  用戶 A（分數較高）→ 可主動發訊息給 用戶 B（分數較低）✅
  用戶 B（分數較低）→ 無法主動發訊息給 用戶 A（分數較高）❌

具體實作邏輯（ChatController@store）：
  sender_score = 發訊息者的 credit_score
  receiver_score = 收訊息者的 credit_score

  if (sender_score < receiver_score - CREDIT_GAP_THRESHOLD)
    return 403 { code: '4032', message: '您的誠信分數不足以向此用戶發送訊息' }
```

> **【待業主確認】** `CREDIT_GAP_THRESHOLD` 的值：
> - 方案 A：任何差距都不能逆向（receiver > sender 就擋）
> - 方案 B：差距 > 30 分才擋（容忍小幅差距）
> - 建議採方案 A，與 xlsx 原始描述一致。

### 8.1 分數區間對應發訊權限表（方案 A）

| 發訊者等級 | 可發訊息給 |
|-----------|---------|
| 頂級（91-100） | 所有等級 |
| 優質（61-90） | 優質、普通、受限 |
| 普通（31-60） | 普通、受限 |
| 受限（1-30） | 受限（同區間）|
| 停權（0） | 無法發訊息 |

### 10.2 credit_score_logs 表

```sql
CREATE TABLE credit_score_logs (
  id            BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  user_id       BIGINT UNSIGNED NOT NULL,
  old_score     TINYINT UNSIGNED NOT NULL,
  new_score     TINYINT UNSIGNED NOT NULL,
  delta         SMALLINT NOT NULL,              -- 正數加分，負數扣分
  type          VARCHAR(50) NOT NULL,           -- 見下方 §10.3
  reason        VARCHAR(255) NOT NULL,          -- 人類可讀說明
  related_id    BIGINT UNSIGNED NULL,           -- 關聯 ID（ticket_id / date_id / report_id）
  operator_id   BIGINT UNSIGNED NULL,           -- 管理員 ID（系統自動為 NULL）
  operator_type ENUM('system', 'admin') NOT NULL DEFAULT 'system',
  created_at    TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_credit_logs_user (user_id, created_at DESC)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 10.3 type 枚舉值

| type 值 | 中文標籤 | Tag 顏色 | 說明 | 分數 |
|---------|---------|---------|------|------|
| `email_verify` | Email 驗證 | green | Email 驗證完成 | +5 |
| `phone_verify` | 手機驗證 | green | 手機驗證完成 | +5 |
| `adv_verify_male` | 男性進階驗證 | green | 男性信用卡驗證（ECPay NT$100）| +15 |
| `adv_verify_female` | 女性進階驗證 | green | 女性照片審核通過 | +15 |
| `date_gps` | 約會驗證（GPS）| green | QR 約會驗證，GPS ≤ 500m | +5 |
| `date_no_gps` | 約會驗證（無GPS）| green | QR 約會驗證，GPS 未通過 | +2 |
| `date_noshow` | 約會爽約 | red | 約會爽約（管理員認定）| -10 |
| `report_submit` | 提交檢舉 | red | 提交/收到檢舉雙方扣分 | -10 |
| `report_result_refund` | 檢舉退分 | blue | 檢舉不成立 / 用戶自取消退還 | +10 |
| `report_result_penalty` | 檢舉處分 | red | 檢舉屬實對被舉方額外處分 | -5 |
| `admin_reward` | 管理員獎勵 | gold | 管理員手動加分（delta > 0）| 可變 |
| `admin_penalty` | 管理員懲罰 | volcano | 管理員手動扣分（delta < 0）| 可變 |
| `content_violation` | 內容違規 | red | 發布不當內容（管理員刪除時扣）| -5 |
| `appeal_refund` | 申訴補分 | blue | 停權申訴核准補分 | 可變 |

**Tag 顏色設計原則：**
- `green`：加分（系統自動觸發）
- `gold`：加分（管理員給予，區隔系統 vs 人為）
- `blue`：中性退還（檢舉/申訴歸還，非獎勵）
- `red`：扣分（系統觸發）
- `volcano`：扣分（管理員給予，深紅視覺警示）

> **規格擴充記錄（2026-04-25）**：
> - 新增 `report_result_penalty`（與 `report_result_refund` 對稱，處理「檢舉屬實額外處分」）
> - 新增 `appeal_refund`（申訴核准補分，語意清晰取代舊的 `appeal_approved`）
> - `admin_reward` / `admin_penalty` 依 delta 符號分路（≥0 用 reward，<0 用 penalty，delta=0 在 controller 層擋下）

### 11.3 Observer 自動停權（DEV-004 已有骨架，此處為完整版）

```php
// app/Observers/UserCreditObserver.php
public function updated(User $user): void
{
    if (!$user->isDirty('credit_score')) return;

    $newScore = $user->credit_score;
    $oldStatus = $user->getOriginal('status');

    // 自動停權
    if ($newScore <= 0 && in_array($oldStatus, ['active', 'restricted'])) {
        $user->updateQuietly(['status' => 'auto_suspended']);
        Cache::put("suspended_user:{$user->id}", true, now()->addYears(1));
        Mail::to($user->email)->queue(new AccountAutoSuspendedMail($user));
        event(new UserAutoSuspended($user));
    }

    // 自動解停（管理員調分觸發）
    if ($newScore >= 30 && $oldStatus === 'auto_suspended') {
        $user->updateQuietly(['status' => 'active']);
        Cache::forget("suspended_user:{$user->id}");
        Mail::to($user->email)->queue(new AccountReactivatedMail($user));
    }
}
```
