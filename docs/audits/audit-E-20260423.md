## Issue E-001

**規格位置：** DEV-008 §4.1（加分事件）、§6（跨功能分數影響矩陣）  
**規格內容：**
```
完成 Email 驗證  → +5（system_settings: credit_add_email_verify）
完成手機驗證     → +5（system_settings: credit_add_phone_verify）
男性信用卡進階驗證 → +15（system_settings: credit_add_adv_verify_male）
觸發時機：各自的 VerificationController 成功回調後
```

**程式碼位置：** backend/app/Http/Controllers/Api/V1/VerificationController.php；backend/app/Http/Controllers/Api/V1/PhoneVerificationController.php；backend/app/Http/Controllers/Api/V1/Admin/VerificationController.php  
**程式碼現況：**
```php
// 全站 grep 'CreditScoreService::adjust' ：
// ❌ VerificationController（Email 驗證）— 無任何 CreditScoreService 呼叫
// ❌ PhoneVerificationController（手機驗證）— 無任何 CreditScoreService 呼叫
// ❌ 男性信用卡進階驗證（ECPay callback）— 無任何 CreditScoreService 呼叫

// ✅ 唯一存在的驗證類加分：
// Admin/VerificationController.php:90
CreditScoreService::adjust($user, 15, 'verification_approved', '女性驗證通過 +15', $adminId);
```

**差異：** Email 驗證（+5）、手機驗證（+5）、男性進階驗證（+15）三個加分事件完全未實作。女性照片驗證（+15）已實作。用戶完成這三種驗證行為，信用分數不會上升。

**等級：** 🟠 High

**建議方案：**
- Option A：在 `VerificationController@verifyEmail` 成功路徑加入 `CreditScoreService::adjust($user, +5, 'email_verify', ...)`；`PhoneVerificationController@confirmPhone` 同樣處理；男性進階驗證在 ECPay notify callback 完成後呼叫
- 注意一次性限制：呼叫前需先確認 `credit_score_histories` 中無同類型記錄

**推薦：** A — 這是規格明確要求的加分機制，直接影響用戶分數成長路徑。

## Issue E-002

**規格位置：** DEV-008 §11.2  
**規格內容：**
```php
// 禁止 hardcode 分數值，一律從 system_settings 讀取
$delta = Cache::remember('setting:credit_add_date_gps', 300, fn() =>
    SystemSetting::where('key', 'credit_add_date_gps')->value('value') ?? 5
);
```

**程式碼位置：** backend/app/Services/DateService.php:149；backend/app/Services/ReportService.php:41；backend/app/Http/Controllers/Api/V1/Admin/VerificationController.php:90  
**程式碼現況：**
```php
// DateService.php:149 — 讀 config()，非 SystemSetting
$score = $gpsVerified
    ? (int) (config('mimeet.credit_add_date_gps', 5))
    : (int) (config('mimeet.credit_add_date_no_gps', 2));

// ReportService.php:41 — hardcode -10
CreditScoreService::adjust($reporter, -10, 'report_filed', '送出檢舉');

// Admin/VerificationController.php:90 — hardcode 15
CreditScoreService::adjust($user, 15, 'verification_approved', '女性驗證通過 +15', $adminId);
```

**差異：** 規格要求全部分值從 `system_settings` 讀取（並加 Redis 快取），後台管理員可即時修改。實際上幾乎所有呼叫點均 hardcode 或讀 `config()`，`system_settings` 的 `credit_add_*`/`credit_sub_*` 鍵值對調整後不會影響任何計算。

**等級：** 🟠 High

**建議方案：**
- Option A：將所有 hardcode 分值替換為 `(int) SystemSetting::get('credit_add_date_gps', 5)` + Redis cache；DEV-008 §11.2 已提供標準寫法

**推薦：** A — 後台 UI「動態調整分值」功能的存在前提是程式碼確實讀取 `system_settings`；目前等同死功能。

## Issue E-003

**規格位置：** DEV-008 §7.2、§11.3  
**規格內容：**
```php
// 自動停權時
Mail::to($user->email)->queue(new AccountAutoSuspendedMail($user));
event(new UserAutoSuspended($user));

// 解停時
Mail::to($user->email)->queue(new AccountReactivatedMail($user));
```

**程式碼位置：** backend/app/Observers/CreditScoreObserver.php  
**程式碼現況：**
```php
public function updated(User $user): void
{
    // 自動停權
    if ($newScore <= 0 && ...) {
        DB::table('users')->...->update(['status' => 'auto_suspended', ...]);
        Cache::put("suspended_user:{$user->id}", true, now()->addYear());
        Log::info("[AutoSuspend] user #{$user->id} ...");
        // ❌ 無 Mail::queue(new AccountAutoSuspendedMail($user))
    }

    // 自動解停
    if ($newScore >= 30 && ...) {
        DB::table('users')->...->update(['status' => 'active']);
        Cache::forget("suspended_user:{$user->id}");
        Log::info("[AutoRestore] user #{$user->id} ...");
        // ❌ 無 Mail::queue(new AccountReactivatedMail($user))
    }
}
```

**差異：** 自動停權和解停時均無 Email 通知。用戶帳號被自動停權（分數歸零）或恢復正常，不會收到任何 Email。

**等級：** 🟠 High

**建議方案：**
- Option A：在 `CreditScoreObserver` 對應分支加入 `Mail::to($user->email)->queue(new AccountAutoSuspendedMail($user))` 及 `AccountReactivatedMail`；需先確認兩個 Mail class 存在
- 注意：`DB::table()->update()` 繞過 Eloquent（避免無限遞迴），`$user->fresh()` 後才能取到最新狀態

**推薦：** A — 停權 Email 是用戶體驗的必要組件；用戶不知道被停權原因將直接流失。

## Issue E-005

**規格位置：** DEV-008 §10.2（欄位定義）、§11.1（呼叫方式）  
**規格內容：**
```sql
-- 規格欄位（§10.2）：
related_id    BIGINT UNSIGNED NULL,    -- 關聯 ID
operator_type ENUM('system', 'admin'), -- 操作者類型

-- 規格呼叫方式（§11.1）：
CreditScoreService::adjust(
    user, delta, reason, type, relatedId, operatorId, operatorType
);
```

**程式碼位置：** backend/database/migrations/2026_04_08_000005_create_credit_score_histories_table.php；backend/app/Services/CreditScoreService.php；backend/app/Models/CreditScoreHistory.php  
**程式碼現況：**
```php

// ❌ 無 related_id
// ❌ 無 operator_type

// CreditScoreService::adjust() 簽名：
public static function adjust(User $user, int $delta, string $type, string $reason, ?int $operatorId = null): void
// ❌ 無 relatedId 參數
// ❌ 無 operatorType 參數
// 注意：欄位名 score_before/score_after（規格是 old_score/new_score）
```

**差異：** `related_id`（關聯 ticket/date/report ID）和 `operator_type`（system vs admin）兩個欄位缺失，無法追溯分數異動的操作來源類型與關聯物件。另外欄位名稱 `old_score`/`new_score` → `score_before`/`score_after` 有細微差異。

**等級：** 🟡 Medium

**建議方案：**
- Option A：更新規格 §10.2 以實際 schema 為準（`score_before`/`score_after`，缺 `related_id`/`operator_type`）
- Option B：新增 migration 加入 `related_id` 和 `operator_type` 欄位，並更新 `CreditScoreService::adjust()` 簽名

**推薦：** A（短期）— 稽核追溯功能對 MVP 不是必須；長期可擇機實作 B。

## Issue E-009

**規格位置：** API-001 §10.8.1（成功回應）  
**規格內容：**
```json
{ "ticket_no": "A202600001" }
```

**程式碼位置：** backend/app/Http/Controllers/Api/V1/AppealController.php:41  
**程式碼現況：**
```php
'ticket_no' => "A{$report->id}",  // 例：A1, A42, A123
// 規格範例：A202600001（零填充到 9 位，包含年份）
```

**差異：** 格式不同：規格示意為年份前綴+序號（零填充），實作為 `A` + 原始 ID（無填充）。功能可用，但若前端依格式解析年份或序號，邏輯會錯誤。

**等級：** 🔵 Low

- Option A：更新規格說明 ticket_no 格式為 `A{id}`（純序號）
- Option B：實作改為 `sprintf('A%09d', $report->id)` 以符合規格格式

**推薦：** A — 規格範例中的格式是示意性的，非強制；以現有格式更新規格即可。
