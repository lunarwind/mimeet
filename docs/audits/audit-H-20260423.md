### Issue H-001 — 舉報 `type` 型別不符（🔴 Critical）

**規格位置：** API-001 §8.1  
**受影響檔案：**
- `frontend/src/api/reports.ts:8` — `type: number`
- `backend/app/Http/Controllers/Api/V1/ReportController.php:24` — validation rule

**現象：** 前端定義 `CreateReportPayload.type: number`，`api/reports.ts` 送出 numeric type（1=騷擾、2=假冒、3=詐騙、4=不雅、5=其他）。後端驗證規則：

```php
'type' => 'required|in:fake_photo,harassment,spam,scam,inappropriate,system_issue,other',
```

數字型別永遠不在字串 enum 內 → `POST /reports` 一律 422 Unprocessable Entity。

**規格原文（§8.1）也使用數字：**
```
type: 1=一般檢舉, 2=系統問題, 3=匿名聊天檢舉, 4=取消訂閱申請
```

但規格的數字與前端的數字語意也不一致（前端有 5 個選項，規格定義 4 個）。

**根本問題：** 三方（規格、前端、後端）type 定義各自獨立演化，互不對齊。

**建議修法（最小改動）：** 後端改用前端送的數字，或後端接受 string，前端改傳 string；需三方對齊後統一。

### Issue H-002 — §6 動態內容 API 完全未實作（🟠 High）

**規格位置：** API-001 §6（動態內容 API）、§6.2（動態留言）  
**受影響範圍：** 後端 routes + 前端 views

**規格要求的端點：**
```
POST   /api/v1/contents              — 發布動態
GET    /api/v1/contents              — 動態牆列表（含篩選）
POST   /api/v1/contents/{id}/like    — 按讚
DELETE /api/v1/contents/{id}/like    — 取消讚
DELETE /api/v1/contents/{id}         — 刪除動態
GET    /api/v1/posts/{id}/comments   — 取得留言（付費功能）
POST   /api/v1/posts/{id}/comments   — 發表留言（付費功能）
DELETE /api/v1/posts/{id}/comments/{comment_id} — 刪除留言
```

**現狀：**

| 層 | 狀態 |
|----|------|
| DB Migration | ✅ `posts`、`post_images`、`post_likes` 表已存在 |
| Backend routes | ❌ `routes/api.php` 無任何 `/contents` 或 `/posts` 路由 |
| Backend controller | ❌ `Controllers/Api/V1/` 無 PostController 或 ContentController |
| Frontend views | ❌ `src/views/app/PostsView.vue` 不存在（DEV-003 目錄規格列有，但實際未建立）|
| Frontend api/ | ❌ `src/api/posts.ts` 不存在 |

**影響：** 動態牆功能完全不可用。DB schema 已準備但應用層（後端 API + 前端 UI）均未建立。

### Issue H-003 — Notification mark-read 未呼叫後端 API（🟠 High）

**規格位置：** API-001 §10.7.3 + §10.7.4  
**受影響檔案：** `frontend/src/views/app/NotificationsView.vue:64–72`

**NotificationsView.vue 的 mark-read 邏輯：**

```javascript
// 單筆點擊（§10.7.3 — 應 PATCH /notifications/{id}/read）
function handleClick(notif: Notification) {
  notif.isRead = true       // ← 只改本地 ref，無 API call
  router.push(notif.actionUrl)
}

// 全部已讀（§10.7.4 — 應 POST /notifications/read-all）
function markAllRead() {
  notifications.value.forEach(n => { n.isRead = true })  // ← 只改本地 ref，無 API call
  uiStore.showToast('已全部標記為已讀', 'success')
}
```

**後端路由存在但未被呼叫：**
```php
Route::patch('notifications/read-all', [NotificationController::class, 'markAllRead']);  // api.php:230
Route::patch('notifications/{id}/read', [NotificationController::class, 'markRead']);    // api.php:231
```

**影響：**
1. 頁面重載後已讀記錄歸零（後端 `is_read` 欄位永不被更新）
2. BottomNav 通知 badge 未讀數不減少（`useNotificationStore.unreadCount` 計算自本地 state，重載後重抓 API 仍顯示全部未讀）
3. 跨裝置同步不可用（後端 unread 狀態從未改變）

### Issue H-004 — §3.6 FCM Token 路由全缺（🟡 Medium）

**規格位置：** API-001 §3.6  
**受影響檔案：** `backend/routes/api.php`（無對應路由）

**規格端點：**
```
POST   /api/v1/me/fcm-token  — 登入或 App 進入前景時呼叫，儲存 FCM Device Token
DELETE /api/v1/me/fcm-token  — 登出時移除 Token
```

**現狀：** `backend/app/Models/FcmToken.php` 存在（model），`fcm_tokens` 表已建立，但 `routes/api.php` 無任何 FCM 相關路由，亦無 FcmTokenController。

**影響：** 前台無法向後端註冊推播 token → Firebase Cloud Messaging (FCM) 推播無法指向指定裝置；現有 `notifications` 表只能在 App 開啟時由 API 拉取，無法主動推播到裝置。

**規格位置：** API-001 §9.1  
**受影響檔案：** 規格文件本身（前後端一致，規格過期）

**評估：** 前後端實作一致，功能可用。規格定義的較複雜版本（認證 + 伺服器端已讀）未建立，但目前方案對公告功能足夠。需更新 §9.1 規格反映實際設計。

### Issue H-008 — §10.7.2 Notification unread-count 專用端點缺失（🔵 Low）

**規格位置：** API-001 §10.7.2  
**受影響檔案：** `backend/routes/api.php`

**規格要求：**
```
GET /api/v1/me/notifications/unread-count → { "data": { "unread_count": 5 } }
```

**現狀：** 無此路由。未讀數透過 `GET /notifications` 列表回應中的 `unread_count` 欄位取得（結合 Issue H-003，此值因 mark-read 未呼叫後端，永遠不準確）。

前端 `useNotificationStore` 的 `unreadCount` 計算自本地 state（`notifications.filter(n => !n.isRead).length`），不呼叫此端點；BottomNav badge 依賴 store。

**影響：** 若未來加入 Web Push 或需快速刷新 badge（不拉全列表），缺少輕量 unread-count 端點。
