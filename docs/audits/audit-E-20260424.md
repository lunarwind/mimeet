## Issue #E-001

**Pass：** P4（業務規則）  
**規格位置：** DEV-008 §11.2 / §4.1 / §5.2  
**規格內容（key 命名範例）：**
```php
SystemSetting::get('credit_add_date_gps', default: 5)   // §4.2
SystemSetting::get('credit_add_email_verify', default: 5) // §4.1
SystemSetting::get('credit_sub_report_user', default: -10) // §5.2
// key 格式：credit_add_* / credit_sub_*（無前綴 "credit_score."）
```
**程式碼位置：** backend/app/Services/CreditScoreService.php:50  
**程式碼現況：**
```php
public static function getConfig(string $key, int $default): int
{
    return (int) SystemSetting::get("credit_score.{$key}", $default);
    // 加了 "credit_score." 前綴 → key 不匹配
}
// 呼叫端：
CreditScoreService::getConfig('email_verified', 5)       // → 讀 credit_score.email_verified
CreditScoreService::getConfig('report_filed_deduct', -10) // → 讀 credit_score.report_filed_deduct
```
**差異說明：** 三層不一致：(1) `getConfig()` 自動加 `credit_score.` 前綴，但 DEV-008 規格說 key 應是 `credit_add_*` / `credit_sub_*`；(2) 呼叫端的短 key 名稱也不符規格（`email_verified` vs `credit_add_email_verify`）；(3) 因此後台管理員即使在 system_settings 正確設定 `credit_add_email_verify` 的值，程式也讀不到，加分值永遠走 hardcode default。DEV-008 §1「可動態調整」設計原則實際上無效。  
**等級：** 🟠 High  
**建議方案：**
- Option A（推薦）：統一改為 `SystemSetting::get("credit_add_{$key}", $default)` 格式，並確認所有呼叫端傳入正確短 key（`email_verified` → `email_verify`）
- Option B：保留現有 `credit_score.` 前綴，但在後台和規格中統一記錄此前綴
**推薦：** A（讓 system_settings 的 key 與 DEV-008 規格一致，使後台調整功能真正生效）

## Issue #E-002

**Pass：** P2（Payload）、P3（回應結構）、P7（前端）  
**規格位置：** docs/API-001 §10.8.1  
**規格內容：**
```
POST /me/appeal
Content-Type: multipart/form-data
- reason: string（必填）
- images[]: file（選填，最多 3 張）

回應: { "data": { "ticket_no": "A202600001", "message": "..." } }
```
**程式碼位置：** frontend/src/api/appeals.ts  
**程式碼現況（前端）：**
```typescript
await client.post('/me/appeal', { reason: payload.reason })
// JSON body，無 FormData，images 欄位從未傳送

export interface AppealResponse {
  ticket_number: string  // ← 規格是 ticket_no
  message: string
}
```
**差異說明：** 兩個問題：(1) 前端以 JSON 發送申訴，後端 validate `images.*` 為 image 檔案，但 JSON 無法傳檔案 → 停權用戶無法上傳佐證截圖；(2) AppealResponse.ticket_number vs 後端回應 ticket_no → 前端無法正確顯示申訴單號。後端 AppealController 實作正確（multipart 驗證），問題在前端未實作 FormData。  
**等級：** 🟠 High  
**建議方案：**
- Option A（推薦）：前端 appeals.ts 改用 FormData，加入 images 上傳支援；同時修正 AppealResponse.ticket_number → ticket_no
**推薦：** A
