## Issue D-001

**規格位置：** API-001 §16.1（⚠️ 實作說明：「目前無統一 /uploads 端點。各類照片使用專用端點上傳。」）  
**規格內容：**
```
§16.1 承認無統一端點；§16.2 描述 DELETE /uploads
前端 useImageUpload.ts 是後續新增的，目標為統一 /uploads
```

**程式碼位置：** frontend/src/composables/useImageUpload.ts:85；frontend/src/views/app/settings/AccountView.vue:317；frontend/src/views/app/ReportsView.vue:12  
**程式碼現況：**
```typescript
// useImageUpload.ts:85 — 所有上傳情境（avatar/profile_photo/report_image）均呼叫此路徑
const res = await client.post<...>('/uploads', formData, { ... })
// ❌ backend/routes/api.php 無任何 Route::post('/uploads', ...) 定義
```
```
AccountView.vue:317 → uploadPhoto(file) → POST /uploads → 404
ReportsView.vue:12  → uploadReport(file) → POST /uploads → 404
```

**差異：** `POST /uploads` 路由不存在。AccountView（個人相冊）和 ReportsView（舉報截圖）透過此 composable 上傳均 404 失敗。直接呼叫 `POST /me/photos`、`POST /me/avatars` 的路由存在，但 `useImageUpload` 沒有對應到它們。

**等級：** 🔴 Critical

**建議方案：**
- Option A：後端新增 `POST /uploads` 路由，依 `context` 欄位分派至對應邏輯（avatar→avatars、profile_photo→photos、report_image→reports）
- Option B：修改 `useImageUpload.ts` 依 `context` 呼叫正確的現有端點（`/me/photos`、`/me/avatars`、`/reports` 圖片欄位）

**推薦：** B — 不需新增後端路由；對應現有端點語義更清晰。但需確認 report_image 的後端上傳端點（`/reports` 是否接受 multipart/form-data）。

```php
Route::prefix('subscriptions')->group(function () {   // 複數
```

## Issue D-006

**規格位置：** API-001 §10.3  
**規格內容：**
```
POST /api/v1/subscriptions/cancel-request
回應 201：
{ "ticket_number": "R2024122000042",
  "subscription_expires_at": "2025-01-19T23:59:59Z",
  "notice": "服務將持續至訂閱到期日，到期後不自動續費" }
```

**程式碼位置：** backend/app/Http/Controllers/Api/V1/SubscriptionController.php（cancelRequest 方法）  
**程式碼現況：**
```php
return response()->json([
    'success' => true,
    'code' => 200,
    'message' => '取消訂閱申請已提交',
], 200);  // ❌ 201 → 200；缺 ticket_number、subscription_expires_at、notice
```

**差異：** HTTP status 201→200；回應 body 缺少 `ticket_number`（工單號）、`subscription_expires_at`（服務持續至）、`notice`（說明文字）。前端若依規格讀這三個欄位，均為 `undefined`，取消確認頁無法顯示到期日。

**等級：** 🟡 Medium

**建議方案：**
- Option A：更新 `cancelRequest()` 回應補上 `subscription_expires_at`（讀 Subscription.expires_at）和 `notice`；`ticket_number` 若無 ticket 系統可省略或生成假 ID
- Option B：更新規格以實作為準，僅記載 `message`

**推薦：** A（部分） — `subscription_expires_at` 對 UX 有意義（告知用戶服務持續至何時），應補上；`ticket_number` 若無 ticket 系統可由規格中移除；`notice` 前端可 hardcode。

## Issue D-009

**規格位置：** API-001 §10.9（PATCH /me/subscription/auto-renew 錯誤回應）  
**規格內容：**
```json
非付費會員 (403)：{ "error": { "code": "4032", "message": "無有效訂閱" } }
```

**程式碼位置：** backend/app/Http/Controllers/Api/V1/SubscriptionController.php（update 方法）  
**程式碼現況：**
```php
if (!$sub) {
    return response()->json([
        'success' => false,
        'code' => 404,
        'message' => '目前沒有有效訂閱',
```

**等級：** 🔵 Low

**建議方案：**
- Option A：更新規格為 404（語義更符合「找不到有效訂閱」）
- Option B：後端改回 403 + error code `4032`

**推薦：** A — 404 語義合理（「您沒有訂閱可設定」比「您沒有權限」更準確）；更新規格即可。
