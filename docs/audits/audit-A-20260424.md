## Issue #A-001

**Pass：** P1（路由層）、P6（Middleware）  
**規格位置：** docs/API-001 §2.2.2.1（行 309）、§2.2.2.2（行 329）  
**規格內容：**
```
POST /api/v1/auth/verify-phone/send
Authorization: Bearer {access_token}   ← 明確標示需登入

POST /api/v1/auth/verify-phone/confirm
Authorization: Bearer {access_token}   ← 明確標示需登入
```
**程式碼位置：** backend/routes/api.php:69-72  
**程式碼現況：**
```php
Route::prefix('auth')->middleware('throttle:otp')->group(function () {
    Route::post('/verify-phone/send', [AuthController::class, 'verifyPhoneSend']);
    Route::post('/verify-phone/confirm', [AuthController::class, 'verifyPhoneConfirm']);
});
```
**差異說明：** 路由群組只套 `throttle:otp`，缺少 `auth:sanctum`。Remote 實測：無 Bearer token 發 POST → 200 Success，驗證碼實際發出。任何人可指定任意手機號碼觸發 OTP SMS，造成 SMS 轟炸/費用濫用風險。  
**等級：** 🔴 Critical  
**建議方案：**
- Option A（推薦）：在路由群組加 `auth:sanctum`，同時維持 `throttle:otp`
  ```php
  Route::prefix('auth')->middleware(['auth:sanctum', 'throttle:otp'])->group(function () {
  ```
- Option B：在 Controller 內用 `auth()->guard('sanctum')->check()` 做軟性拒絕（較弱）
**推薦：** A（路由層保護最乾淨，符合規格 Authorization 標示，且不影響已登入的正常流程）

## Issue #A-002

**Pass：** P3（回應結構）、P7（前端 API 層）  
**規格位置：** docs/API-001 §2.1.1（行 167-183）  
**規格內容：**
```json
{
  "success": true,
  "code": 201,
  "message": "註冊成功，請查收驗證郵件",
  "data": {
    "user": { "status": "pending_verification", "created_at": "..." },
    "verification": { "email_sent": true, "expires_at": "2024-12-20T11:30:00Z" }
  }
}
```
**程式碼位置：** backend/app/Http/Controllers/Api/V1/AuthController.php:124-143  
**程式碼現況：**
```php
return response()->json([
    'success' => true,
    'code' => 'REGISTER_SUCCESS',   // 字串非整數 201
    'data' => [
        'user' => [..., 'status' => $user->status],  // 'active' 非 'pending_verification'
        'token' => $token,           // 規格無 token 欄位
        // 缺 'verification' block
    ],
], 201);
```
**差異說明：** 三個差異：(1) `code` 是字串 `'REGISTER_SUCCESS'` 不是整數 `201`；(2) 回應有 `token`，規格無；(3) 缺少 `verification.email_sent` 和 `verification.expires_at`。前端 RegisterView.vue:108 已適配使用 token，但規格合約不一致。  
**等級：** 🟠 High  
**建議方案：**
- Option A：更新規格 §2.1.1，加入 token 欄位，補充 verification 為 optional
- Option B：改程式碼回傳 verification block，移除 token（需更新前端）
- Option C（推薦）：更新規格 §2.1.1 以反映實際回應（token 是設計決定，非失誤）；補充 `code` 為字串型態
**推薦：** C（前後端已依賴 token 欄位，改程式碼成本高；更新規格成本低）

## Issue #A-003

**Pass：** P10（跨模組副作用）  
**規格位置：** docs/API-001 §2.1.1（行 183）  
**規格內容：**
```json
"user": { "status": "pending_verification" }
```
**程式碼位置：** backend/app/Http/Controllers/Api/V1/AuthController.php:108  
**程式碼現況：**
```php
$user = User::create([
    ...
    'status' => 'active',   // 規格要求 'pending_verification'
]);
```
**差異說明：** 新用戶一建立就是 `active` 狀態，email 尚未驗證。登入邏輯（line 211）只檢查 `suspended/auto_suspended`，不阻擋 `active` 用戶。這意味著未驗證 email 的用戶可以立即登入並使用所有已登入功能。規格設計中 `pending_verification` 應限制功能存取。  
**等級：** 🟠 High  
**建議方案：**
- Option A（推薦）：更新規格 §2.1.1，標記此為「已知差異」並說明系統採 email_verified 欄位而非 status 作為驗證閘
- Option B：將 register 的 status 改為 `pending_verification`，login 加入狀態檢查（大改動，影響現有用戶）
**推薦：** A（系統已採用 email_verified boolean 欄位管控驗證狀態，status 的語意已重新定義；改 B 風險大）

## Issue #A-004

**Pass：** P2（請求 Payload）  
**規格位置：** docs/API-001 §2.1.1（行 163-164）  
**規格內容：**
```json
{ ..., "group": 2, ... }
```
**程式碼位置：** backend/app/Http/Controllers/Api/V1/AuthController.php:31-41（formatValidator）  
**程式碼現況：** `group` 欄位不在 validate rules 中，也不在 `User::create()` 中；前端 RegisterView.vue 亦未傳送此欄位

**等級：** 🟡 Medium

- Option A（推薦）：規格備注此欄位為「Phase 2 待實作」，暫緩
- Option B：實作後端驗證 `group: 0|1|2` + User::create 加入 group 欄位
**推薦：** A（確認業務需求後再實作，避免引入未定義行為）

## Issue #A-005

**Pass：** P2（請求 Payload）  
**規格位置：** docs/API-001 §2.1.1（行 161）  
**規格內容：**
```json
{ "password_confirmation": "SecurePassword123", ... }
```
**程式碼位置：** backend/app/Http/Controllers/Api/V1/AuthController.php:31-41  
**程式碼現況：**
```php
$formatValidator = Validator::make($input, [
    'password' => ['required', 'string', 'min:8'],
    // 缺少 'password' => ['confirmed'] 或 'password_confirmation' 規則

```
**差異說明：** register 的 validate rules 不含 `password_confirmation` 驗證，用戶可以傳入不一致的 password 和 password_confirmation 而後端不報錯。（`changePassword` 函式有 confirmed rule，但 `register` 無）  
**等級：** 🟡 Medium  
**建議方案：**
- Option A（推薦）：在 register 的 formatValidator 加入 `'password' => ['required', 'string', 'min:8', 'confirmed']`
- Option B：移除規格中的 password_confirmation 欄位要求
**推薦：** A（加一個 rule 即修復，無副作用）

## Issue #A-006

**Pass：** P5（錯誤碼）  
**規格位置：** docs/API-001 §12（錯誤碼規範）  
**規格內容：**
```json
{ "1001": "invalid_credentials", "1003": "account_suspended", "1010": "verification_code_expired" }
```
**程式碼位置：** backend/app/Http/Controllers/Api/V1/AuthController.php（多處）  
**程式碼現況：**
```php
// login 回傳：
'error' => ['code' => 'INVALID_CREDENTIALS', ...]   // 字串
// verifyPhoneConfirm 回傳：
'error' => ['code' => '1023', ...]                   // 數字字串
// resetPassword 回傳：
'error' => ['code' => '1010', ...]                   // 數字字串（符合規格）
```
**差異說明：** 部分端點（login, logout, verifyPhoneSend）使用語意字串碼（`INVALID_CREDENTIALS`, `PHONE_CODE_SENT`），部分（resetPassword, verifyPhoneConfirm）使用數字字串碼（`'1010'`, `'1023'`）。規格統一定義為數字碼 1001-1010。格式不一致，前端難以統一處理。  
**等級：** 🟡 Medium  
**建議方案：**
- Option A：規格 §12 更新，承認兩種格式並為字串碼建立對照表
- Option B（推薦）：統一改用字串語意碼（可讀性佳，不易衝突），並更新規格
**推薦：** B（字串碼已較廣泛使用，且可讀性優於數字碼；只需更新規格 §12 的定義方式）

## Issue #A-007

**Pass：** P4（業務規則）  
**規格位置：** docs/API-001 §2.1.2（行 214）  
**規格內容：** 規格未明確定義 login 的 throttle 機制  
**程式碼位置：** backend/routes/api.php:54  
**程式碼現況：**
```php
Route::post('/login', [AuthController::class, 'login']);  // 無 throttle middleware
```
**差異說明：** register 用 `throttle:register`，forgot-password 用 `throttle:otp`，但 login 無任何 throttle middleware。Controller 內部實作了手動 Cache lockout（email 5次/5min，IP 20次/5min），功能上有保護，但不走 Laravel 統一的 throttle 機制，且無 Retry-After header。  
**等級：** 🔵 Low  
**建議方案：**
- Option A（推薦）：加入 `throttle:login`，同時保留 Controller 的細粒度 lockout
- Option B：維持現狀，在規格 §2.1.2 補充「由 Controller 自行處理限流」說明
**推薦：** A（一致性較好，且 Laravel throttle 會回傳標準 `Retry-After` header）
