### 2.1.2 用戶登入 `POST /auth/login`

**請求參數差異：**

| 欄位 | 規格 | 後端實作 | 前端 `LoginPayload` |
|------|------|---------|---------------------|
| `email` | 必填 | ✅ | ✅ |
| `password` | 必填 | ✅ | ✅ |
| `remember_me` | 選填 | 🔵 接受但無用（Sanctum token 無 TTL 差異）| 🔵 未送出 |
| `device_info` | 選填物件 | 🔵 未使用 | 🔵 未送出 |

**回應結構差異（Critical）：**

規格 §2.1.2 定義：
```json
"tokens": {
  "access_token": "...",
  "refresh_token": "...",
  "token_type": "Bearer",
  "expires_in": 3600
}
```

實際回傳：
```json
"token": "eyJ0eXAiOiJKV1Qi..."  // 純字串，無 refresh_token，無 expires_in
```

前端 `LoginResponse` 介面已對齊實際（`data.token: string`），**與規格不同但前後端已自洽**。

**限流差異：**

| 限制機制 | 規格說明 | 實作 |
|---------|---------|------|
| 路由 throttle middleware | 規格未明訂 | ❌ 無 throttle middleware 在 login 路由 |
| Controller 層鎖定 | 規格未提及 | ✅ 5次 Email 失敗鎖定5分鐘；20次 IP 失敗鎖定5分鐘 |

**等級：** 🟠 High（`tokens` vs `token` 結構差異若未來規格文件被新開發者參考，會產生誤導；refresh token 根本不存在）

### 2.2.2 手機驗證（兩段路由）

#### `POST /auth/verify-phone/send`

**Auth middleware 差異：**

| 項目 | 規格 | 實作 |
|------|------|------|
| `Authorization: Bearer` | ✅ 規格明訂需要 | ❌ **後端路由無 `auth:sanctum`** |
| `throttle:otp` | ✅（規格提及限流）| ✅ 路由已掛 `throttle:otp` |

**後端路由：** `Route::post('auth/verify-phone/send', ...)->middleware('throttle:otp')`  
**缺少 `auth:sanctum`，未登入用戶可呼叫此端點。**

**等級：** 🟡 Medium（設計上刻意讓未登入用戶可發送？或遺漏？需確認業務規則）

#### `POST /auth/verify-phone/confirm`

**錯誤碼差異：**

| 情境 | 規格錯誤碼 | 實作錯誤碼 |
|------|-----------|-----------|
| OTP 錯誤或過期 | `OTP_INVALID` | `1021`（OTP 不存在）/ `1022`（OTP 錯誤）/ `1023`（OTP 過期）|

規格使用一個統一代碼，實作分為三種細分代碼。前端目前只判斷 HTTP status 200/422，未解析 error code，所以無影響。

**等級：** 🔵 Low（文件不一致，功能正常）

## 4. 問題彙整與建議行動

### 🔴 Critical（0 項）

無高危安全漏洞。

---

### 🟠 High（3 項）

#### H-1：`POST /auth/refresh` — 規格定義但完全未實作

**現象：** 規格 §2.1.3 定義 refresh token 端點，但後端無此路由，前端無此函式，Sanctum Personal Access Token 模式本身也沒有 refresh 機制。

**影響：** 誤導讀者以為有 token 輪換機制。若未來要遷移至 JWT 方案，有誤解風險。

**建議：**
- **短期：** 將 §2.1.3 標記為「Phase 2 / 暫不實作」或整節刪除。
- **長期：** 如需 token 輪換，改用 Sanctum expiration（`config/sanctum.php` 的 `expiration`）+ 讓前端在 401 時重新登入。

---

#### H-2：`POST /auth/refresh` 延伸 — 登入回應 `tokens` vs `token`

**現象：** 規格 §2.1.2 定義 `data.tokens.{ access_token, refresh_token, token_type, expires_in }`，實際回傳 `data.token`（純字串）。

**影響：** 任何使用規格書接入的外部系統（如 App 原生端）會因欄位路徑錯誤而無法取得 token。

**建議：** 更新 §2.1.2 登入成功回應，改為：
```json
"data": {
  "user": { ... },
  "token": "eyJ0eXAiOiJKV1Qi..."
}
```
並移除 `refresh_token`、`expires_in`（Sanctum PAT 無此概念）。

---

#### H-3：`POST /auth/credit-card-verification` — 未實作

**現象：** 規格 §2.2.4 定義男性信用卡驗證端點，後端無路由，前端無函式。

**建議：** 確認此功能所在 Sprint；若尚未排入開發計畫，在規格書標記狀態 `[未實作，Phase N]`。

---

### 🟡 Medium（3 項）

#### M-1：`POST /auth/verify-phone/send` 缺少 `auth:sanctum`

**現象：** 規格 §2.2.2.1 要求 `Authorization: Bearer {access_token}`，後端路由只掛 `throttle:otp`，未登入用戶可呼叫。

**可能意圖：** 允許未完成 Email 驗證但已發 token 的用戶使用？或允許未登入狀態直接驗證？

**建議：** 與 PM 確認業務需求，若確實需要驗證才能使用，補上 `auth:sanctum`。若刻意放開（如允許未登入先驗手機），更新規格書說明。

---

#### M-2：進階驗證路徑在 §2.2.3 與 §16.3 互相矛盾

**現象：**
- §2.2.3 記載：`POST /auth/photo-verification/start`、`POST /auth/photo-verification/upload`
- §16.3/§16.4 記載（較新）：`POST /me/verification-photo/request`、`POST /me/verification-photo/upload`

**建議：** 將 §2.2.3 的舊路徑更新為 §16.3/§16.4 的實際路徑，並刪除重複說明。

---

#### M-3：`POST /auth/reset-password` 無 throttle

**現象：** 後端路由無 throttle middleware。

**風險評估：** 低（token 為 64 字元隨機 hex，暴力破解需 2^256 次嘗試，理論安全）。

**建議：** 加上 `throttle:otp` 做縱深防禦，防止日誌爆炸與 DB 查詢浪費。

---

### 🔵 Low（2 項）

#### L-1：Register 請求缺少合規欄位驗證

**現象：** `group`、`terms_accepted`、`privacy_accepted`、`anti_fraud_read` 規格定義為必填，但後端未驗證，前端未送出。

**建議：** 法規合規角度（個資法、服務條款確認），建議後端補上 `required|boolean|accepted`，前端補送對應欄位（RegisterView Step 2 已有勾選 UI，只差 payload 未送出）。

---

#### L-2：verify-phone/confirm 錯誤碼文件不一致

**現象：** 規格 §2.2.2.2 說錯誤碼為 `OTP_INVALID`，實際為 `1021`/`1022`/`1023`。

**建議：** 更新規格書列出三個細分代碼；或後端統一改為字串錯誤碼（符合其他端點慣例）。
