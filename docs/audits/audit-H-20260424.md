## Issue #H-001

**Pass：** P10（跨模組副作用）  
**規格位置：** docs/API-001 §10.7 / §11.4a（通知依賴 FCM 推播）  
**程式碼位置：** backend/app/Services/FcmService.php  
**程式碼現況：**
```php
public function send(string $token, string $title, string $body, array $data = []): bool
{
    if (config('app.env') !== 'production') {
        Log::info('[FCM STUB]', compact('token', 'title', 'body', 'data'));
        return true;  // Stub: 非 production 只 log
    }
    // NOTE: 生產環境需實作 FCM HTTP v1 API
    // $serverKey = config('services.fcm.server_key');
    return false;  // ← Production 永遠返回 false，push 不發出
}
```
Remote .env: `FCM_SERVER_KEY=`（空值）  
**差異說明：** Production 環境下 FcmService::send() 直接 return false，FCM HTTP v1 API 呼叫從未實作（被 comment 掉）。`FCM_SERVER_KEY` 為空。結果：站內通知（notifications 表）正常寫入，但行動裝置推播（FCM）完全無效。用戶在手機鎖屏時收不到任何通知（新訊息、約會邀請等）。  
**等級：** 🔴 Critical  
**建議方案：**
- Option A（推薦）：實作 FCM HTTP v1 API：取得 OAuth 2.0 Service Account 憑證，使用 `google/auth` 套件發送推播
- Option B（短期）：使用第三方 FCM 套件（如 `kreait/laravel-firebase`）並設定 Firebase Service Account JSON
**推薦：** A 或 B，需先取得 Firebase 專案 Service Account 憑證

## Issue #H-007

**Pass：** P6（認證）、P8（前端 UI）  
**規格位置：** docs/API-001（隱含：各功能需 Lv2 驗證）  
**程式碼位置：** frontend/src/router/guards.ts / frontend/src/router/routes/app.ts  
**程式碼現況：**
```typescript
// app.ts 定義了 minLevel meta：
{ path: 'messages', meta: { requiresAuth: true, minLevel: 2 } },
{ path: 'dates', meta: { requiresAuth: true, minLevel: 2 } },

// guards.ts 完全沒有讀取 minLevel：
// 5. Suspended → /suspended
// 6. Not verified → /login
// ← 無 minLevel 檢查！
```
**差異說明：** routes 定義了 `minLevel: 2` 要求但 guards.ts 不讀取此 meta，Lv0/Lv1 用戶可直接導航到 /app/messages 和 /app/dates。後端 API 仍有 `middleware:membership:2` 保護（不會真的能發訊息），但前端體驗差：用戶看到 UI 卻被 API 拒絕，無法得到明確引導到升級頁面。  
**等級：** 🟠 High  
**建議方案：**
- Option A（推薦）：在 guards.ts 加入 minLevel 檢查：
  ```typescript
  const minLevel = (to.meta.minLevel as number) ?? 0
  if (auth.user && auth.user.membership_level < minLevel) {
    return { path: '/app/shop' }
  }
  ```
**推薦：** A

{

## Issue #H-008

**Pass：** P3（回應結構）、P4（業務規則）  
**規格位置：** docs/API-001 §8.1（POST 用字串 type；GET 用數字 type）  
**規格內容（不一致性）：**
```
POST /reports 送出時用字串：type: "harassment" | "scam" | ...
GET  /reports 列表回傳用數字：type: 1 | 2 | 3
GET  /reports query 也用數字：type: 1|2|3
```
**程式碼位置：** backend/app/Http/Controllers/Api/V1/ReportController.php  
**程式碼現況：** store() 正確驗證字串 type。但 reports 表是否存儲字串還是整數？history() 直接 proxy 到 index()。List 回傳時 type 欄位是什麼型態需確認。  
**差異說明：** 規格本身自相矛盾：submit 用語意字串，list 用數字 enum。前端需要一套 type → label 對照表處理兩種格式。這可能是舊版規格（數字）和新版設計（字串）混用造成的遺留問題。  
**等級：** 🟡 Medium  
**建議方案：**
- Option A（推薦）：統一改為字串（與 POST submit 一致），更新規格 §8.1.2 和 §10.4 的 query/response 格式
**推薦：** A
