## Issue C-001

**規格位置：** API-001 §4.1.3  
**規格內容：**
```
GET /chats/{id}/messages
Query: before_id (int) — 取此 ID 以前的訊息
Response: { data: { messages: [...], next_before_id: int | null } }
```

**程式碼位置：** backend/app/Http/Controllers/Api/V1/ChatController.php（messages 方法）；frontend/src/api/chat.ts  
**程式碼現況：**
```php
// ChatController::messages()
$messages = $query->cursorPaginate(30);
// 回應含 next_cursor（不是 next_before_id）
```
```typescript
// frontend/src/api/chat.ts — fetchMessages()
// 呼叫時無分頁參數（cursor/before_id 皆未傳）
await axios.get(`/chats/${conversationId}/messages`)
```

**差異：** 分頁機制完全不同。規格為 keyset（`before_id` 整數）；實作為 cursor-based（opaque cursor string）。前端目前未傳任何分頁參數，僅取第一頁。

**影響：** 前端無法向上載入舊訊息（分頁功能失效）；規格與實作的 next page token 欄位名稱不同，若前端依規格讀 `next_before_id` 會拿到 `undefined`。

**建議：** 以實作為準，將規格 §4.1.3 更新為 cursor 分頁；同步更新前端 `fetchMessages()` 加入 `cursor` 參數支援。

## Issue C-002

**規格位置：** API-001 §4.1.6  
**規格內容：**
```
PATCH /chats/{id}/messages/read
Body: { "data": { "message_ids": [101, 102, 103] } }
Response: { "success": true }
```

**程式碼位置：** backend/routes/api.php；backend/app/Http/Controllers/Api/V1/ChatController.php（markRead 方法）；frontend/src/api/chat.ts  
**程式碼現況：**
```php
// routes/api.php
Route::patch('/chats/{conversation}/read', [ChatController::class, 'markRead']);
// 注意：路徑是 /read，不是 /messages/read

// ChatController::markRead()
// 對整個 conversation 標記已讀，不接受 message_ids
$conversation->messages()->where('sender_id', '!=', $userId)->update(['is_read' => true]);
```
```typescript
// frontend/src/api/chat.ts
markConversationRead: (id) => axios.patch(`/chats/${id}/read`)
// ✅ 前端與實作一致（但與規格不同）
```

**差異：** 兩點不符：(1) URL 路徑 `/chats/{id}/read` vs 規格 `/chats/{id}/messages/read`；(2) 實作為對話層級批次標記，規格為指定 `message_ids[]` 的精細控制。

**影響：** 若依規格傳 `message_ids`，後端忽略此參數，改為對話全部標記，語義不同。前端已與實作對齊，實際功能可用；但第三方若依規格開發將呼叫錯誤端點。

**建議：** 以實作為準更新規格，移除 `message_ids`，改記載對話層級標記行為。

## Issue C-004

**規格位置：** API-001 §5.1（回應）  
**規格內容：**
```json
{
  "data": {
    "invitation_id": 1,
    "qr_code": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
  }
}
```

**程式碼位置：** backend/app/Http/Controllers/Api/V1/DateController.php（store 方法回應）  
**程式碼現況：**
```php
// DateController::store() 回應

    'success' => true,
    'data' => [
        'date_invitation' => $invitation,
```

**影響：** 前端若讀 `data.qr_code` 得到 `undefined`；QR Code 產生邏輯（JWT vs hex）與規格描述不同。

**建議：** 更新規格 §5.1 回應為 `qr_token`（hex），並補充完整的回應物件結構。

## Issue C-006

**規格位置：** API-001 §4.2.1（MessageSent 事件 payload）  
**規格內容：**

```json
  "id": 101,
  "conversation_id": 1,
```

**程式碼位置：** backend/app/Events/ChatMessageSent.php（broadcastWith 方法）

```php
public function broadcastWith(): array
{
    return [
```

## Issue C-009

```php
// 2. 發送時間 5 分鐘內
// 3. 對方尚未讀取（is_read = false）
```

**差異：** 規格未記載撤回限制（5 分鐘、對方未讀）和 `RECALL_DENIED`（422）錯誤情境。

**影響：** 前端可能在超時或對方已讀後仍顯示撤回按鈕，點擊後無提示（靜默失敗）。

**建議：** 在規格 §4.1.5 補充撤回條件說明與 `RECALL_DENIED` 422 錯誤碼。
