## Issue #C-001

**Pass：** P4（業務規則）  
**規格位置：** docs/API-001 §4.1.3（行 ~1096）  
**規格內容：**
```
> - Lv0–Lv2.x（未付費）用戶每日上限 30 則訊息
> - Lv3（付費會員）無上限
```
**程式碼位置：** backend/app/Services/ChatService.php:checkDailyLimit()  
**程式碼現況：**
```php
$limits = [
    0   => 5,    // Lv0 — 只有 5 則（規格寫 30）
    1   => 30,   // Lv1
    1.5 => 100,  // Lv1.5（規格未定義此等級）
    2   => 30,   // Lv2
];
```
**差異說明：** 兩個問題：(1) Lv0 使用者每日只能發 5 則訊息，規格說應為 30 則；(2) Lv1.5（女性驗證帳號）每日可發 100 則，此等級在 §4.1.3 完全未記載。這兩個差異都直接影響用戶體驗，且無法從規格推斷。  
**等級：** 🟠 High  
**建議方案：**
- Option A：確認業務需求後更新規格 §4.1.3，補充各 Membership Level 的實際上限表格
- Option B：將 Lv0 改為 30 以符合規格（若規格是正確的）
**推薦：** A（Lv0=5 可能是刻意的業務決定以激勵用戶驗證；規格應反映實際的分級設計）

```json
"last_message": {

  "content": "你好",

  "sender_id": 456,
```

## Issue #C-004

**Pass：** P1（路由層）  
**規格位置：** docs/API-001 §5.1.1（行 ~1290）, §5.1.4/5.1.5（行 ~1370）  
**規格內容：**
```
POST /api/v1/date-invitations     # 創建（§5.1.1）
PATCH /api/v1/dates/{id}/accept   # 接受（§5.1.4）
POST /api/v1/dates/verify         # QR 驗證（§5.2.1）
```
**程式碼位置：** backend/routes/api.php:141-154  
**程式碼現況：**
```php
// /date-invitations 前綴 (DateInvitationController)
Route::post('/', ...)          // POST /date-invitations
Route::get('/', ...)           // GET /date-invitations
Route::patch('/{id}/response')  // PATCH /date-invitations/{id}/response（規格名）
Route::post('/verify', ...)    // POST /date-invitations/verify（重複！）

// /dates 前綴 (DateController)  
Route::post('/', ...)          // POST /dates（規格無此端點）
Route::patch('/{id}/accept')   // PATCH /dates/{id}/accept（規格名）
Route::patch('/{id}/decline')  // PATCH /dates/{id}/decline（規格名）
Route::post('/verify', ...)    // POST /dates/verify（重複！）
```
**差異說明：** 存在兩套路由：`/date-invitations`（legacy，nested data 格式）和 `/dates`（新版，flat 格式）。兩者有重疊功能：(1) 創建邀請有兩個端點，payload 格式不同（`scheduled_at` vs `date_time`，`location` vs `location_name`）；(2) QR 驗證有兩個端點（`/date-invitations/verify` 和 `/dates/verify`），兩者功能相同；這造成規格、前端實作、後端維護三方都需要知道用哪個路由。  
**等級：** 🟡 Medium  
**建議方案：**
- Option A（推薦）：廢棄 `/date-invitations/*` 路由，統一使用 `/dates/*`，更新規格 §5 的路由路徑
- Option B：保留兩套，在規格中明確標示 legacy vs current
**推薦：** A（減少維護負擔，統一欄位名稱）

## Issue #C-005

**Pass：** P3（回應結構）、P5（錯誤碼）  
**規格位置：** docs/API-001 §5.2.1（驗證失敗回應，行 ~1440）  
**規格內容：**
```json
{
  "success": false,
  "code": 400,
  "message": "約會驗證失敗",
  "error": {
    "type": "verification_failed",
    "reason": "location_mismatch",
    "details": {
      "current_distance": 600.5,
      "max_allowed_distance": 500,
      "time_window_valid": true,
      "location_valid": false
    }
  }
}
```
**程式碼位置：** backend/app/Http/Controllers/Api/V1/DateController.php:verify()  
**程式碼現況：**
```php
return response()->json([
    'success' => false,
    'error' => [
        'code' => $e->getMessage(),    // 字串，如 'TOKEN_EXPIRED'
        'message' => $e->getMessage(), // 重複
    ],
], $httpCode);
```
**差異說明：** 回應格式極簡，不含 `type`、`reason` 或 `details` 物件。GPS 驗證失敗時前端無法獲知當前距離（`current_distance`）或允許距離（`max_allowed_distance`），無法對用戶顯示有意義的距離提示（如「您距離約會地點 600 公尺，超出允許距離 500 公尺」）。  
**等級：** 🟡 Medium  
**建議方案：**
- Option A（推薦）：在 DateService::verifyQrToken 計算距離後，將結果傳回 Controller 以填充 details 物件
- Option B：更新規格 §5.2.1 簡化錯誤回應定義（移除 details）
**推薦：** A（對 UX 有實際幫助）
