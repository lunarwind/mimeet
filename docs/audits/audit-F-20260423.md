## Issue F-001

**規格位置：** API-002 §10（RBAC 角色權限）  
**規格內容：**
```
11 個 permission key；各端點標注所需權限（members.view / members.edit / reports.process / chat.view 等）
角色：super_admin（全部）/ admin（多數）/ cs（僅報告+會員查看）
```

**程式碼位置：** backend/app/Http/Kernel.php:67；backend/routes/api.php:238-355  
**程式碼現況：**
```php
// Kernel.php:67 — middleware 已存在
'admin.permission' => \App\Http\Middleware\CheckAdminPermission::class,

// routes/api.php — 所有 admin 路由
Route::middleware(['admin.auth', 'admin.log'])->group(function () {
    // ❌ 無任何 ->middleware('admin.permission:members.view') 等呼叫
    Route::get('/members', [...]);        // 規格要求 members.view
    Route::get('/payments', [...]);       // 規格要求 payments.view
    Route::get('/chat-logs/search', [...]);// 規格要求 chat.view
    // ...所有端點均無 admin.permission 保護
});

// 唯一的例外：super_admin only 路由
Route::middleware('check.super_admin')->prefix('settings')->group(function () { ... });
```

**差異：** `CheckAdminPermission` middleware 已完整實作（含 DB 查詢、super_admin wildcard、錯誤回應），且已在 Kernel 註冊，但**從未掛載至任何路由**。結果：`cs` 角色的管理員可直接呼叫 `GET /payments`、`GET /chat-logs/search`、`DELETE /members/{id}` 等其沒有權限的端點，系統完全不阻擋。

**等級：** 🔴 Critical

**建議方案：**
- Option B：在各路由加入 `->middleware('admin.permission:members.view')` 等；建議使用 `Route::group` 集中套用相同 permission 的端點

```php
// 範例修法
Route::middleware('admin.permission:members.view')->group(function () {
    Route::get('/members', ...);
    Route::get('/members/{id}', ...);
    Route::get('/members/{id}/chat-logs', ...);
});
Route::middleware('admin.permission:members.edit')->group(function () {
    Route::patch('/members/{id}/actions', ...);
    Route::patch('/members/{id}/permissions', ...);
    // ...
});
```

**推薦：** B — middleware 已準備好，只需在路由層掛載。

## Issue F-002

**規格位置：** API-002 §2.2（GET /auth/me）、§2.3（POST /auth/logout）  
**規格內容：**
```
GET  /api/v1/admin/auth/me      → 200 { admin: { id, name, email, role, permissions } }
POST /api/v1/admin/auth/logout  → 200 { success: true }
```

**程式碼位置：** backend/routes/api.php:238-355  
**程式碼現況：**
```php
Route::prefix('admin')->group(function () {
    Route::post('/auth/login', ...);   // ✅ 存在

    Route::middleware(['admin.auth', 'admin.log'])->group(function () {
        // ❌ GET /auth/me — 不存在
        // ❌ POST /auth/logout — 不存在
        Route::get('/members', ...);  // 下一個存在的路由
    });
});
```

**差異：** `GET /auth/me` 和 `POST /auth/logout` 兩個端點均不存在。Admin 前端登入後無法查詢自身 token 是否有效（page refresh 後會話驗證無依據）；也無法透過 API 登出（token 無法被 revoke）。

**等級：** 🟠 High

**建議方案：**
- Option B：新增 `GET /auth/me`（讀 `$request->user()`）和 `POST /auth/logout`（`$request->user()->currentAccessToken()->delete()`）到 admin 路由群組

**推薦：** B — 這兩個是基礎 auth 功能，admin SPA refresh 或 logout 都依賴這兩個端點。

## Issue F-003

**規格位置：** API-002 §4.3（PATCH /members/{id}/actions）  
**規格內容：**
```
action 可選值：
  adjust_credit（調整分數，params: value + reason）
  set_level（調整等級，params: level + reason）
  require_reverify（要求重驗，params: verify_type + reason）
  suspend（停權，params: reason）
  unsuspend（解停，params: reason）
  add_note（備註，params: note）
  delete（刪除，params: reason，requires members.delete）
```

**程式碼位置：** backend/app/Http/Controllers/Api/V1/AdminController.php:259  
**程式碼現況：**
```php
'action' => 'required|string|in:adjust_score,suspend,unsuspend,verify_phone,unverify_phone,verify_advanced,unverify_advanced',
// ❌ adjust_credit → adjust_score（名稱不同，前端若依規格送 adjust_credit 得 422）
// ❌ set_level     — 完全未實作
// ❌ require_reverify — 完全未實作
// ❌ add_note      — 完全未實作
// ❌ delete        — 獨立路由 DELETE /members/{id}，不是 actions 的一部分
// ✅ suspend / unsuspend — 存在
// ➕ verify_phone / unverify_phone / verify_advanced / unverify_advanced — 額外實作（規格未列）
```

**差異：** action 名稱 `adjust_credit`→`adjust_score` 不同，前端依規格呼叫會 422。`set_level`（手動調整會員等級）、`require_reverify`（要求重新驗證）、`add_note`（管理員備註）三個 action 未實作。`delete` 以獨立路由實作（合理）。

**等級：** 🟠 High

**建議方案：**
- Option C：(1) 在 validation `in:` 加入 `adjust_credit` alias（或改名）；(2) 實作 `set_level` 直接操作 `membership_level`；(3) 實作 `add_note` 寫入 `admin_notes`；(4) `require_reverify` 可暫標記用戶需重驗狀態

**推薦：** C — 至少需修正 `adjust_credit` alias，`set_level` 和 `add_note` 是高頻後台操作。

## Issue F-011

**規格位置：** API-002 §2.1（登入回應）  
**規格內容：**
```json
{
  "data": {
    "token": "...",
    "token_type": "Bearer",
    "expires_in": 28800,
    "admin": { "id", "name", "email", "role", "permissions": ["*"] }
  }
}
```

**程式碼位置：** backend/app/Http/Controllers/Api/V1/AdminController.php:21  
**程式碼現況：**
```php
'data' => [
    'admin' => ['id', 'email', 'name', 'role'], // ❌ 無 permissions
    'token' => $token,
    // ❌ 無 token_type
    // ❌ 無 expires_in
],
```

**差異：** 回應缺少 `token_type`（前端自行 hardcode `Bearer`）、`expires_in`（前端無法知道 token 有效期）、`admin.permissions`（前端無法依權限控制 UI 顯示）。

**等級：** 🔵 Low

**建議方案：**
- Option B：在 login 回應加入 `token_type: 'Bearer'`、`expires_in: 28800`、`permissions`（讀 `admin_role_permissions` 或 `['*']` for super_admin）

**推薦：** B — `permissions` 對前端動態隱藏無權限功能有實際意義，應補上。
