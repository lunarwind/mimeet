## Issue #B2-001

**規格位置：** docs/API-001_前台API規格書.md §3.2.1（搜尋業務規則）
**規格內容：** `privacy_settings.show_in_search = false` 的用戶不出現在搜尋
**程式碼位置：** `backend/app/Http/Controllers/Api/V1/UserController.php:113-117`
**程式碼現況：**
```php
$query->where(function ($q) {
    $q->whereNull('privacy_settings')
      ->orWhereRaw("JSON_EXTRACT(privacy_settings, '$.show_in_search') != 'false'");
});
```
**差異說明：**
`JSON_EXTRACT` 的型別比較脆弱。當 `privacy_settings` 非 NULL 但 `show_in_search` 鍵缺失時，`JSON_EXTRACT` 返回 NULL，MySQL 中 `NULL != 'false'` 是 NULL（非 TRUE），導致該用戶被**錯誤排除**。
另：比較 JSON boolean `false` 與字串 `'false'` 依賴 MySQL 隱含型別轉換。
**等級：** 🟡 Medium
**建議方案：**
- Option A：改用 `JSON_VALUE(privacy_settings, '$.show_in_search')` + 明確值比較
- Option B：改用 Laravel 的 `whereJsonContains` / `whereJsonDoesntContain`
- Option C（最保守）：`JSON_EXTRACT(privacy_settings, '$.show_in_search') IS NULL OR JSON_EXTRACT(privacy_settings, '$.show_in_search') != cast(FALSE as JSON)`
**推薦：** B（`whereJsonDoesntContain`）

## Issue #B2-003

**規格位置：** API-001 §10.2（誰來看我）
**規格內容：** 訪客列表呈現訪問過我的用戶
**程式碼位置：** `backend/app/Http/Controllers/Api/V1/UserController.php:589-651`
**程式碼現況：**
```php
$query = UserProfileVisit::where('user_profile_visits.visited_user_id', $userId)
    ->join('users', 'users.id', '=', 'user_profile_visits.visitor_id')
    // 無 WHERE users.status = 'active' 過濾
```
**差異說明：**
已停權（`status=suspended/auto_suspended`）或已刪帳（`deleted_at IS NOT NULL`）的用戶若在停權前造訪過，仍會出現在訪客列表中（甚至 join 到已軟刪除用戶）。規格未明文要求過濾，但顯示已停權用戶的資料可能造成 UX 問題。
**等級：** 🔵 Low
**建議：** 加入 `->where('users.status', 'active')->whereNull('users.deleted_at')` 過濾
