**等級：** 🟠 High  
**建議方案：**
- Option A（推薦）：實作 `GET /api/v1/users/recommendations`，至少返回依誠信分數排序的推薦清單（不需要 ML 算法，先用規則）

**推薦：** A（推薦功能是核心約會 app 體驗，但若資源不足先標 Phase 2 可接受）

## Issue #B-002

**Pass：** P3（回應結構）  
**規格位置：** docs/API-001 §3.1.5（`GET /api/v1/me/settings` 成功回應）  
**規格內容：**
```json
{
  "data": {
    "membership": {
      "is_paid": true,
      "expires_at": "2025-02-20T10:30:00Z",
      "days_remaining": 15
    }
  }
}
```
**程式碼位置：** backend/app/Http/Controllers/Api/V1/UserController.php:settings()  
**程式碼現況：** `settings()` 函式返回 `profile / account / verification / privacy_settings` 四個區塊，**完全沒有 `membership` 區塊**。`auth/me` 端點有回傳 subscription 資料，但 settings 沒有。

**等級：** 🟠 High  
**建議方案：**
- Option A（推薦）：在 `settings()` 加入 membership block（可從 Subscription 模型查詢，同 `auth/me` 的實作邏輯）
**推薦：** A（`auth/me` 已有相同查詢邏輯，複製過來即可）

## Issue #B-003

**Pass：** P1（路由）、P4（業務規則）、P9（邊界）  
**規格位置：** docs/API-001 §3.1.3（行 ~195）、§3.1.4（行 ~222）、§3.1 照片業務規則（行 ~240）、§3.3（行 ~450）  
**規格內容（§3.1 照片系統）：**
```
POST /api/v1/users/{user_id}/photos        # 上傳照片
DELETE /api/v1/users/{user_id}/photos/{photo_id}  # 刪除照片
PUT /api/v1/me/photos/{photo_id}/set-avatar       # 設為頭像
規則：相冊最少 2 張 (error 2032)、最多 6 張 (error 2031)
```
**規格內容（§3.3 頭像槽位）：**
```
無獨立 /me/photos 端點；上限 3 張；以 JSON 儲存 avatar_slots
```
**程式碼位置：** backend/routes/api.php:85-89  
**程式碼現況：**
```php
Route::post('/me/photos', [UserController::class, 'uploadPhoto']);  // §3.3 說「無此端點」
// 無 DELETE /users/{id}/photos/{photo_id}
// 無 PUT /me/photos/{id}/set-avatar
// uploadPhoto() 回應缺少 id / thumbnail_url / order / status 欄位
// 無 2031/2032 min/max count 驗證
```
**差異說明：** §3.1 與 §3.3 直接矛盾（§3.3 明確說「無獨立 `/me/photos` 端點」但 §3.1 定義了它；§3.1 說上限 6 張但 §3.3 說 3 張）。Code 實作了 §3.3 的槽位系統，但同時保留了 §3.1 的 `/me/photos` 路由。缺少照片 delete by ID 路由和 min/max count 驗證。  
**等級：** 🟡 Medium  
**建議方案：**
- Option A（推薦）：確認業務方向後修正規格內部矛盾：
  - 若採 §3.3 槽位系統：移除 §3.1 的 photos 端點定義（或標為廢棄），移除 `/me/photos` 路由，補充 2031 槽位滿錯誤碼
  - 若採 §3.1 完整相冊：實作 delete by ID、min/max 驗證、uploadPhoto 完整回應
**推薦：** A-i（採 §3.3 槽位系統，較簡潔）
