## Issue B-001

**規格位置：** API-001 §3.3（§3.1.3、§3.1.4 亦相關）  
**規格內容：**
```
GET  /me/photos             取得相冊（回傳 photos[]、max_photos: 6）
POST /me/photos             新增相冊照片（先上傳取 URL，再呼叫本端點）
PATCH /me/photos/sort       排序（body: { order: [3,1,2] }）
DELETE /me/photos/{photo_id} 刪除（相冊至少需保留 2 張，否則 422 error 2032）
```

**程式碼位置：** backend/routes/api.php:72-89  
**程式碼現況：**
```php
// 存在的路由
Route::post('/me/photos', [UserController::class, 'uploadPhoto']);   // ✅
Route::get('/me/avatars', [UserController::class, 'getAvatarSlots']); // 不同系統
Route::post('/me/avatars', [UserController::class, 'uploadAvatar']); // 不同系統
Route::patch('/me/avatars/active', [UserController::class, 'setActiveAvatar']);
Route::delete('/me/avatars', [UserController::class, 'deleteAvatar']);
// ❌ GET /me/photos — 不存在
// ❌ PATCH /me/photos/sort — 不存在
// ❌ DELETE /me/photos/{id} — 不存在
```

**差異說明：** 規格設計的是「相冊照片」系統（最多 6 張，有排序），但實作是「頭像槽位」系統（`avatar_slots` JSON 欄位，最多 3 張，無 photo_id 概念）。兩者架構根本不同：相冊系統需有獨立 photo 表，槽位系統只是 JSON 陣列。前端 AccountView 已使用頭像槽位 API（`/users/me/avatars`）實作頭像切換，但相冊照片管理（ProfileView 展示、相冊瀏覽）功能缺失。

**等級：** 🟠 High  
**建議方案：**
- Option A：改規格，將 §3.3 改寫為頭像槽位系統說明（`GET /users/me/avatars` 等現有端點）
- Option B：實作完整相冊系統（需新增 `user_photos` 表 + 遷移）
- Option C：雙向修改，在規格書承認兩套機制並存

**推薦：** A（短期）+ B（長期）。先更新規格書對齊現況，避免文件誤導；中長期在 Sprint 排入相冊功能。

## Issue B-003

**規格位置：** API-001 §3.6  
**規格內容：**
```
POST   /me/fcm-token   { token: "...", platform: "web|ios|android" }
DELETE /me/fcm-token   { token: "..." }
```

**程式碼位置：** backend/routes/api.php  
**程式碼現況：** 無任何 `/me/fcm-token` 路由。全域搜尋無 `FcmController` 或 `fcm_token` 相關邏輯。

**差異說明：** FCM Token 端點未實作，代表後端無法針對特定裝置送出行動推播通知（APNs/FCM）。站內通知仍可透過 WebSocket（Reverb）推送，但用戶離線時（App 在背景）無法收到推播。

**等級：** 🟠 High  
**建議方案：**
- Option A：改規格，標記為 Phase 2
- Option B：實作 FCM Token 儲存（新增 `user_fcm_tokens` 表 + controller）

**推薦：** A（若行動 App 非當前 Sprint 範圍），B（若已在推行動 App）

## Issue B-006

**規格位置：** API-001 §3.2.1（第 260-330 行）  
**規格內容（相關部分）：**
```
GET /users/search?verified_only=true&sort=credit_score&sort_direction=desc
業務規則：30天未登入自動隱藏（last_login_at < NOW() - 30 days）
```

**程式碼位置：** backend/app/Http/Controllers/Api/V1/UserController.php:73-244  
**程式碼現況：**
```php
$request->validate([
    // 無 verified_only, sort, sort_direction, include_inactive 參數
]);
// 無 30 天未登入過濾
// 排序固定：資料完整度 DESC → credit_score DESC → last_active_at DESC
```

**差異說明：** ① 缺少 `verified_only` 參數（規格重要業務需求：只看已驗證用戶）。② 排序無法由前端指定（規格支援 `sort` + `sort_direction`）。③ 30 天未登入用戶仍出現在搜尋結果，違反規格業務規則（廢棄帳號影響用戶體驗）。

**等級：** 🟡 Medium  
**建議方案：**
- Option B：改程式碼，補 `verified_only` 過濾；補 30 天未登入過濾（`last_active_at > now()->subDays(30)` 或 status 管理）

**推薦：** B。`verified_only` 過濾容易實作（一行 where）；30 天隱藏規則應在 search() 補上。

## Issue B-010

**規格位置：** API-001 §3.1.2  
**規格內容：** `PATCH /users/{user_id}` 請求欄位 `introduction`、`job`、`preferences`  
**程式碼位置：** backend/app/Http/Controllers/Api/V1/UserController.php:32-63  
**程式碼現況：** 路由為 `PATCH /users/me`；接受欄位為 `bio`（非 `introduction`）、`occupation`（非 `job`）；無 `preferences` 物件。  

**差異說明：** 規格請求欄位名與實作不同。前端 AccountView 已手動補 mapping（`bio: form.introduction, occupation: form.job`），功能正常。`UpdateProfilePayload` type 在 `types/user.ts` 使用規格名（`introduction`/`job`），但 PATCH 呼叫時實際發送 `bio`/`occupation`，兩處不一致可能混淆後續開發者。

**等級：** 🔵 Low  
**建議方案：**
- Option A：更新規格 §3.1.2 為實際欄位名（`bio`/`occupation`）
- Option C：統一為規格名（需改 DB column 或 mapping 全鏈路）

**推薦：** A（更新文件），並將 `types/user.ts` 的 `UpdateProfilePayload` 改為和 PATCH payload 一致。

## Issue B-011

**規格位置：** API-001 §3.1.5（業務規則，相冊相關）與 §3.1.1  
**差異說明：** error code `2031` 在兩處被使用：
1. §3.3.2 定義 `2031` = 相冊最多 6 張照片（`POST /me/photos` 超限）
2. UserController `show()` line 262 使用 `2031` = 無法查看此用戶（被封鎖）

Error code 命名空間衝突，會造成前端無法僅依 code 判斷錯誤類型。

**等級：** 🔵 Low  
**建議方案：** 更正封鎖查看的 error code 為未使用的值（如 `2035` 或更具語義的字串 `USER_BLOCKED`）。

## 附記：PATCH 路由設計差異（非 Issue）

規格 §3.1.2 定義 `PATCH /users/{user_id}`，實作為 `PATCH /users/me`。兩者在語義上完全等效（用戶只能改自己的資料），不算真正的 Bug，但破壞了 REST 一致性。此差異已在 B-010 一併記錄。
