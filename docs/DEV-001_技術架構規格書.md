## 1. 專案概述

MiMeet 是一個針對台灣市場的高端包養交友平台，核心特色為：
- **誠信分數系統**：動態評估用戶可信度，控制平台互動權限
- **QR碼約會驗證**：創新的線下見面驗證機制
- **多重身份驗證**：Email / 手機 / 進階（照片/信用卡）三層驗證
- **匿名聊天室**：第二階段功能，提供匿名公開互動空間

### 1.1 開發範疇

| 範疇 | 說明 |
|------|------|
| 前台 Web | Vue.js 3 SPA，面向一般用戶（甜爹/甜心/經紀公司） |
| 後台管理 | React 18 SPA，面向平台管理員（super_admin / admin / cs） |
| 後端 API | Laravel 10 RESTful API + WebSocket |
| 資料庫 | MySQL 8.0（主要資料）+ Redis 7（快取/會話/佇列） |
| 儲存 | DigitalOcean Spaces（圖片/文件），本地 CDN 加速 |

## 6. 安全架構

### 6.1 認證機制（v1.2 更新 — Multi-Guard 架構）

系統採用**前後台身份完全物理隔離（Multi-Guard Architecture）**，前台用戶與後台管理員儲存於不同資料表，使用不同的 Sanctum Guard，Token 權限互不相通。

#### 6.1.0 Guard 對照表

| Guard | Provider | Model | 資料表 | 用途 |
|-------|----------|-------|--------|------|
| `sanctum` | `users` | `App\Models\User` | `users` | 前台 SPA 用戶認證 |
| `admin` | `admins` | `App\Models\AdminUser` | `admin_users` | 後台管理員認證 |

```php
// config/auth.php
'guards' => [
    'sanctum' => ['driver' => 'sanctum', 'provider' => 'users'],
    'admin'   => ['driver' => 'sanctum', 'provider' => 'admins'],
],
'providers' => [
    'users'  => ['driver' => 'eloquent', 'model' => User::class],
    'admins' => ['driver' => 'eloquent', 'model' => AdminUser::class],
],
```

#### 6.1.1 各場景認證機制

| 場景 | Guard | 機制 |
|------|-------|------|
| 前台 SPA 用戶 | `auth:sanctum` | Sanctum Cookie（SPA 模式）+ CSRF Token |
| 後台管理員 | `auth:admin` | Sanctum Bearer Token（abilities: `['admin']`） |
| WebSocket 連線 | `auth:sanctum` | Sanctum Token 認證 |
| 第三方 Webhook（綠界回呼） | 無 Guard | CheckMacValue SHA256 簽章驗證 |
| 綠界發票 API 呼叫 | 無 Guard | AES-128-CBC 加密（Data 欄位） |

#### 6.1.2 隔離效果

- **前台 Token 無法存取後台 API**：`auth:admin` middleware 僅認 `admin_users` 表核發的 Token，前台用戶 Token 自動回傳 401。
- **後台 Token 無法存取前台 API**：`auth:sanctum` middleware 僅認 `users` 表核發的 Token，後台管理員 Token 自動回傳 401。
- **管理員使用前台功能**：管理員若需測試前台功能，需以前台流程註冊獨立帳號。可透過後台 `PATCH /admin/members/{id}/permissions` 手動調整該帳號之會員等級（Lv0-3）、誠信分數（0-100）與帳號狀態，所有調整均寫入 `credit_score_histories` 與 `admin_operation_logs`。

### 6.1.3 金流與發票架構（v1.1 新增）

依據綠界官方最新技術文件直接串接，**完全不依賴第三方開源套件**（如 ecpay-sdk）：
- **金流 API**：全方位金流 AioCheckOut/V5（[官方文件 v2509](https://developers.ecpay.com.tw/2509/)）
- **發票 API**：B2C 電子發票 Issue（[官方文件 v7809](https://developers.ecpay.com.tw/7809/)）

**核心機制：**

| 項目 | 實作方式 |
|------|----------|
| 金流 CheckMacValue | 手工實作 SHA256（ksort → HashKey/IV 包夾 → urlencode → .NET 相容轉換 → 小寫 → SHA256 → 大寫） |
| 發票資料加密 | AES-128-CBC（PKCS7 padding）：urlencode → 加密 → Base64 |
| 發票回應解密 | Base64 → AES-128-CBC 解密 → urldecode |
| 動態環境切換 | `system_settings` 表儲存 `ecpay.mode`（sandbox / production），後台 UI 即時切換 |
| 金鑰管理 | `system_settings` 表儲存（優先）→ config/services.php → .env fallback |

**對帳機制：** 訂單資料表 `orders` 記錄完整對帳欄位：

| 欄位 | 來源 | 用途 |
|------|------|------|
| `ecpay_trade_no` | ECPay 回調 `TradeNo` | 綠界交易序號，對帳用 |
| `ecpay_payment_date` | ECPay 回調 `PaymentDate` | 實際付款時間 |
| `ecpay_payment_type` | ECPay 回調 `PaymentType` | 付款方式（Credit / ATM 等） |
| `invoice_no` | 發票 API `InvoiceNo` | 電子發票號碼 |
| `invoice_date` | 發票 API `InvoiceDate` | 發票開立時間 |
| `invoice_random_number` | 發票 API `RandomNumber` | 發票隨機碼（4 碼） |

後台管理介面 `/admin/payments` 可查看上述所有欄位，點擊「查看詳情」顯示完整對帳資訊。

### 6.3 資料安全

- **傳輸加密**：全站 HTTPS / TLS 1.3
- **密碼儲存**：bcrypt（cost factor 12）
- **敏感欄位加密**：手機號碼、信用卡末四碼使用 AES-256-CBC（Laravel Crypt）
- **JWT / Sanctum Token 有效期**：access token 24h，refresh token 30d
- **Rate Limiting**：登入 API 每 IP 每分鐘 5 次，一般 API 每用戶每分鐘 60 次
- **SQL 注入防護**：Eloquent ORM 參數綁定，禁止原始 SQL 字串拼接
- **XSS 防護**：Response 資料全面 htmlspecialchars，Blade 範本自動 escape
- **CSRF 防護**：前台 SPA 使用 Sanctum Cookie 模式，自動 CSRF 驗證

### 6.3.1 資料保留與銷毀機制（v1.5 新增）

系統實作**兩階段銷毀機制（Two-Phase Destruction）**：

```
第一階段：軟刪除 / 隔離（用戶操作觸發）
  ├─ 訊息刪除 → SoftDeletes（deleted_at 標記，前端不顯示）
  ├─ 照片刪除 → 檔案從 uploads/ 移至 quarantine/{date}/（路徑變更，不物理刪除）
  └─ 帳號刪除 → 7 天冷靜期 → 去識別化（email/phone/nickname 清除）+ 檔案移入隔離區

第二階段：物理銷毀（排程 Job 執行）
  └─ ProcessGdprDeletions（每日排程）
      ├─ 超過 data_retention_days 的軟刪除訊息 → forceDelete()
      ├─ 超過 data_retention_days 的隔離區檔案 → 永久刪除
      └─ 超過 data_retention_days 的用戶活動日誌 → 清除
```

**保留天數配置：**

| 項目 | 前端宣告（隱私權政策） | 實際保留天數 | 配置方式 |
|------|---------------------|-------------|---------|
| 已刪除訊息 | 30 天 | `data_retention_days`（預設 180 天） | 後台 SystemSetting |
| 隔離區照片 | 30 天 | `data_retention_days`（預設 180 天） | 後台 SystemSetting |
| 帳號冷靜期 | 7 天 | 固定 7 天 | 程式碼硬編碼 |
| 驗證照片 | 1 年 | 1 年 | 法規要求 |

> **設計原則：** 前端隱私權政策對用戶宣告最短 30 天保留期，實際保留天數由後台 `data_retention_days` 參數控制（預設 180 天），確保平台具備足夠的安全審核與法遵調閱時間。

**媒體隔離區（Quarantine）：**
- 儲存位置：`Storage::disk('public')` → `quarantine/{YYYY-MM-DD}/{原始路徑}`
- 按日期建立子目錄，方便依時間批次清理
- 排程 Job 掃描日期目錄，超過保留天數則整個目錄 `deleteDirectory()`

**用戶活動審計日誌（UserActivityLog）：**
- 資料表：`user_activity_logs`（user_id, action, metadata JSON, ip_address, user_agent）
- 追蹤事件：login、profile_update、photo_upload/delete、phone_changed、verification_submitted
- 後台 UI：`/user-activity-logs`（super_admin 限定），支援 user_id 搜尋 + action 篩選
